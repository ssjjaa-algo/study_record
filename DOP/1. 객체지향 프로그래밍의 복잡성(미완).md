## OOP 특징 및 시스템 복잡도에 미치는 영향

| **`특징`** | **`복잡도에 미치는 영향`** |
| --- | --- |
| 코드와 데이터가 섞여 있다 | 클래스가 많은 관계에 연루되는 편 |
| 객체가 변경 가능하다 | 코드를 읽을 때 더 많은 생각이 필요
다중 스레드 환경에서 명시적 동기화 필요 |
| 데이터가 객체에 묶여 있다 | 데이터 직렬화가 쉽지 않다 |
| 코드는 클래스에 묶여 있다 | 클래스 계층 구조가 복잡하다 |
- JSON 직렬화와 역직렬화를 다루기 위해 계속해서 클래스 추가가 필요
- OOP에서는 데이터가 클래스에 정의된 경직된 규격을 따라야 한다
    - 데이터가 클래스의 멤버로 고정된다

## 복잡한 클래스 계층 구조

- OOP에서 동일한 코드를 반복해 작성하지 않는 한 가지 방법은 상속
    - 기반 클래스에서 공통 행위를 상속받아 클래스를 만드는 식으로 클래스 위계 구조 설계
- 요구사항이 추가될 때마다 곤란을 겪는다
    - 이미 완성된 Member에 VipMember, SuperMember.. 등등이 계속 추가되는 경우
- Tip: 설계에 클래스 상속 대신 **`구성(composition)`**을 선택

## 이 장을 읽고 난 후기

- 데이터가 클래스의 멤버로, 코드가 클래스에 **`‘구속’`**된다고 표현한다 → 복잡도를 증가시킨다
- api가 추가될 때마다 Dto → Result.. 등의 반복은 꽤나 귀찮은 것이다
