# 상속 관계 매핑

- RDB에는 상속 개념이 없다.
- 슈퍼타입 - 서브타입 관계 모델링 기법이 상속 개념과 가장 유사
- 객체의 상속 구조와 데이터베이스의 슈퍼타입, 서브타입 관계를 매핑하는 것
- 3가지 방법이 존재
    - 각각의 테이블로 변환
        - 각각을 모두 테이블로 만들고 조회할 때 JOIN
    - 통합 테이블로 변환
        - 테이블 하나에 모두 넣기
    - 서브타입 테이블로 변환
        - 서브  타입마다 하나의 테이블을 만든다.

## 조인 전략

- 타입을 구분하는 컬럼이 필요하다. 테이블은 타입의 개념이 없기 때문
    - @Inheritance(strategy = InheritanceType.JOINED)
        - 부모 클래스에 사용, 매핑 전략을 지정해야 함
    - @DiscriminatorColumn(name = “DTYPE”)
        - 부모 클래스에 구분 컬럼 지정
    - @DiscriminatorValue(”M”)
        - 엔티티를 저장할 때 구분 컬럼에 입력할 값 저장
- 장점
    - 테이블 정규화
    - 외래 키 참조 무결성 제약조건 활용
    - 저장공간 효율성
- 단점
    - 조인 多 → 성능 저하
    - 조회 쿼리 복잡
    - INSERT SQL 2번 작용

## 단일 테이블 전략

- 구분 컬럼(DTYPE)으로 어떤 자식 데이터가 저장되었는지 구분.
- 자식 엔티티가 매핑한 컬럼은 모두 null을 허용해야 한다.
- 구분 컬럼을 반드시 사용해야 한다
    - @DistcriminatorColumn
- @Inheritance(startegy = InheritanceType.SINGLE_TABLE)
- 장점
    - 조인이 없어서 조회 성능 빠름
    - 조회 쿼리 단순
- 단점
    - 자식 엔티티 컬럼 모두 null 허용해야함
    - 테이블 사이즈 커질 수 있어서 상황에 따라 조회 성능이 느려질수도 있음e

## 구현 클래스마다 테이블 전략

- 자식 엔티티마다 테이블을 만든다
- 자식 테이블 각각에 필요한 컬럼, 즉 extends한 컬럼들이 모두 있다.
- @Inheritance(startegy = InheritanceType.TABLE_PER_CLASS)
- 장점
    - 서브 타입 구분해서 처리할 때 효과적
    - not null 제약 조건
- 단점
    - 여러 자식 테이블 함께 조회할 때 UNION
    - 자식 테이블을 통합해서 쿼리하기 어렵다

# 복합 키와 식별 관계 매핑

## 식별 관계 vs 비식별 관계

- 식별 관계
    - 부모 테이블의 기본 키를 내려받아서 자식 테이블의 기본 키 + 외래 키
- 비식별 관계
    - 부모 테이블의 기본 키를 받아서 자식 테이블의 외래키로만 사용
    - 필수적 비식별 관계
        - 외래키에 NULL 허용 x, 연관관계를 반드시 맺어야 한다
