# 트랜잭션 격리수준

## 개념

- 동시에 여러 트랜잭션이 처리될 때, 트랜잭션끼리 얼마나 서로 고립되어 있는지를 나타낸다.
    - 특정 트랜잭션이 **`다른 트랜잭션에서 변경한 데이터`**를 볼 수 있게 할 것인가? 말 것인가?

## 4가지 수준

- READ UNCOMMITED
- READ COMMITED
- REPEATABLE READ
- SERIALIZBLE

- mysql은 기본적으로 REPEATABLE READ

## READ UNCOMMITED

- 어떤 트랜잭션의 내용이 commit이나 rollback에 관계 없이 다른 트랜잭션에서 볼 수 있다.
    - X 트랜잭션이 a의 나이를 10살에서 11살로 변경하고 커밋을 하지 않았는데
    - Y 트랜잭션이 a의 나이를 조회할 때 11살로 보인다.
        - **`Dirty read`**
    - X 트랜잭션이 문제가 생겨서 rollback했다
    - 그래도 Y 트랜잭션은 a의 나이를 11살로 생각하고 일한다

## READ COMMITED

- 커밋되야만 다른 트랜잭션에서 조회 가능하다.
    - X 트랜잭션이 a의 나이를 10살에서 11살로 바꾸고 커밋 안했다
    - Y 트랜잭션이 a의 나이를 조회할 때 10살이라 나온다
- 온라인 서비스에서 가장 많이 선택하는 격리수준
- **`NOT-REAPEATABLE READ`** 발생 가능성 존재
    - X 트랜잭션이 a의 나이를 10살에서 11살로 바꾼다
    - Y 트랜잭션 시작
    - Y 트랜잭션이 a의 나이를 조회할 때 10살이라 나온다
    - X 트랜잭션이 커밋 후 종료
    - Y 트랜잭션이 a의 나이를 조회할 때 11살이라 나온다
    - Y 트랜잭션이 종료한다
        - 트랜잭션이 종료되지 않은 상태에서 SELECT의 결과를 보장하지 못하는 상태
            - **`REPEATABLE READ 정합성 보장 x`**

## REPEATABLE READ

- 트랜잭션이 시작되기 전에 커밋한 내용만 조회할 수 있다.
- 트랜잭션이 시작하고 종료될 때까지 조희는 계속해서 같은 값을 반환한다.
- INSERT는 가능하다.
- 데이터를 변경하려면 **`UNDO 영역에 저장해두고 변경한다`**
- **`NOT-REAPEATABLE READ`** 발생하지 않음
    - X 트랜잭션이 a의 나이를 10살에서 11살로 변경
    - Y 트랜잭션이 a의 나이를 조회할 때 10살이라 나온다
    - X 트랜잭션이 커밋한다
    - Y 트랜잭션이 a의 나이를 조회할 때 10살이라 나온다 **`(언두 영역에 백업된 데이터)`**

<aside>
💡 자신의 트랜잭션 번호보다 낮은 번호에서 COMMIT한 것은 undo 영역에서 읽는다.

</aside>

- PHANTOM READ 문제 발생
    - 한 트랜잭션 내에서 같은 쿼리를 두 번 실행했는데, 유령 레코드가 두 번째에서 나타난다.

## SERIALIZABLE (LEVEL 3)

- 모든 동작이 직렬화
- 다른 트랜잭션에서 **수정/삭제/삽입이 불가능**
- 동시성 저하, 성능 하락
- **Dirty Read, Non Repeatable Read, Phantom Read** 문제가 발생 X
- 가장 강한 격리수준
