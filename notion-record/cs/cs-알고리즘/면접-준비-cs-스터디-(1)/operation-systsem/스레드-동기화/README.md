# 스레드 동기화의 필요성

## 멀티태스킹

> **다수**의 작업을 **동시**에 실행시키는 응용프로그램 작성 기법
> 

**장점** : 응용프로그램의 실행 시간을 단축하거나 사용자에 대한 응답 시간을 낮출 수 있다.

**단점** : 다수의 작업들이 공유 데이터에 **동시** 접근 시, 공유 데이터가 훼손되어 예상치 못한 문제가 발생할 수 있다.

## 공유 데이터 동시 접근 문제

### 공유 집계판 문제 : 실생활 예시

1. **학생 A**가 먼저 방에 들어와 집계판의 수 **50**에 10을 더한다.
2. **학생 A**가 계산하고 있는 사이 **학생 B**가 같은 방에 들어온다.
3. 아직 계산되지 않은 집계판의 수 **50**을 보고 그에 10을 더한다.
4. **학생 B**가 집계판에 **60**을 적고 나간다.
5. **학생 A**도 집계판에 **60**을 적고 나간다. **오류!**
6. 집계판에는 학생 A의 값과 학생 B의 값을 더한 70이 적혀 있어야 하나, 60이 적혀있는 오류가 발생

⇒ 2명 이상의 학생이 **동시**에 공유 집계판을 수정하는 상황에 아무 대책이 없었기 때문

### 공유 데이터 동시 접근 문제

```
학생 A 				: Thread A
학생 B 				: Thread B
공유 집계판 		: 공유 변수
학생들의 연산 	: 공유 변수 조작 코드
```

Thread A가 공유 변수에 대한 연산 실행 중 컨텍스트 스위칭 되어,

Thread B가 스케줄되어 공유 변수에 대한 연산 실행 후

다시 Thread A로 컨텍스트 스위칭 되어 연산 이후의 코드부터 수행

(Thread A의 정보를 저장해놓은 레지스터들이 모두 CPU에 복귀)

⇒ **문제의 핵심** : 여러 스레드가 공유 변수에 접근하려고 **경쟁**(race)하려고 하는 데 있다.

## 임계구역

<aside>
💡 **스레드 동기화란?**
공유 데이터에 접근하고자 **경쟁**하는 다수의 스레드가 있는 경우 한 스레드가 공유 데이터를 접근할 때 **배타적**으로 사용할 수 있도록, 다른 스레드가 접근하지 못하도록 **상호 협력**(coordination)하는 것

</aside>

### 임계구역 (Critical Section)

> **공유 데이터**에 접근하는 프로그램 **코드**들
> 
- 임계구역 **설정**은 프로그램 개발자의 판단에 따라 이루어진다.
- 개발자는 ****스스로 **공유 데이터**를 액세스하는 코드 블록을 **임계구역**으로 판단하고, 임계구역에 대한 **상호 배제**가 이루어지도록 작성

# 상호 배제 (Mutual Exclusion)

> 멀티스레드가 실행되는 환경에서 한 스레드가 **임계구역**을 진입하면 **배타적**으로 실행되도록 보장하는 **기법**
> 

## 상호 배제를 포함 프로그램 구조

```
… 	// 일반 코드 (non-critical code)
**entry 코드 (임계구역 진입 코드)**
…		// 임계구역 코드 (critical code)
**exit 코드 (임계구역 진출 코드)**
…		// 일반 코드 (non-critical code)
```

### 일반 코드 (non-critical code)

공유 데이터를 액세스하지 않는 코드 부분

### 임계구역 코드

- 공유 데이터를 접근하는 코드를 **포함**
- 한 스레드에 의해서만 **배타적**으로 사용되도록 보장되어야 하는 코드
- 한 스레드만 사용할 수 있으므로 개발자는 임계구역을 공유 데이터를 액세스하는 **최소한**의 코드로 작성하도록 해야 한다.

### 임계구역 진입 코드 (entry code)

- 임계구역에 진입하기 **전** 필요한 코드
- 다른 스레드가 임계구역에 들어가 있는지 **검사**
    - 임계구역에 들어가 있는 스레드가 없는 경우 다른 스레드가 들어오지 못하도록 **조치**
    - 이미 임계구역에 들어가 있는 스레드가 있는 경우 해당 스레드가 임계구역을 벗어날 때까지 현재 스레드를 **대기**

### 임계 구역 진출 코드 (exit code)

- 임계구역의 실행을 **마칠 때** 필요한 코드
- 다른 스레드가 임계구역에 **진입**할 수 있도록 entry code에서 취한 조치를 풀어놓는 코드

## 상호 배제 구현 방법

<aside>
📌 소프트웨어적 방법, 하드웨어적 방법이 있으나 소프트웨어적 방법은 실제 구현에 있어 여러 문제를 노출하기 때문에 오늘날에는 대부분 **하드웨어적 방법**을 사용

</aside>

### 인터럽트 서비스 금지

> 임계구역을 실행 중인 동안만 입출력 장치나 타이머가 **인터럽트**를 걸지 못하도록 무시 후 임계구역이 끝난 후에 처리
> 
- 인터럽트를 서비스하지 못하도록 하는 기계어 명령 실행
- 스레드가 임계구역 실행 중 선점(preemption)되지 않는다.
- 대표적인 x86 명령으로 cli와 sit가 있다.
    - **cli** : CPU의 **인터럽트 플래그**(IF)를 0으로 리셋 시켜 인터럽트가 발생해도 CPU가 인터럽트 서비스 루틴으로 점프하지 않고 현재 작업을 계속 **수행**하게 한다.
    - **sti** : CPU의 **인터럽트 플래그**(IF)를 1로 설정하여 인터럽스 서비스 루틴을 수행하게 한다.
- **단일 CPU 시스템**에서는 활용 가능하나, 멀티코어를 비롯한 **다중 CPU 시스템**에서는 활용 불가
    - 다른 코어의 인터럽트 서비스까지 금지시킬 수 없기 때문이다.

### 원자 명령(atomic instruction) 사용

> 상호 배제를 위해 특별히 설계된 **CPU 명령**으로, 오늘날 대부분의 컴퓨터 시스템에서 사용
> 
- CPU마다 명령이 서로 다르다.
- lock 변수의 값을 읽어 들이는 명령과 lock 변수에 0 혹은 1을 저장하는 명령 사이 컨텍스트 스위칭이 일어나지 않도록 두 명령을 **하나의 명령**으로 만드는 것
- TSL (Test and Set Lock) 명령이라고도 부른다.

# 멀티스레드 동기화 기법

- **락(lock) 방식** : 뮤텍스(Mutex), 스핀락(Spinlock)
- **wait-signal 방식** :  세마포어(Semaphore)

## 뮤텍스 (Mutex)

> 잠김(locked) / 열림(unlocked) 중 한 상태를 가지는 **락 변수**를 이용하여 오직 한 스레드만이 자원을 **배타적**으로 사용하는 기법이자 소프트웨어 모듈
> 

### 락 변수

- 스레드가 공유 자원을 사용해야 할 시 락 변수에 **잠금**을 뜻하는 **true** 값을 저장하고 임계구역 **진입**
- 반대로 임계구역 코드를 끝마쳤을 시 락 변수에 **false** 값을 저장하고 임계구역 **진출**
- 락을 **true**로 만든 오직 한 스레드만이 해당 공유 자원을 독점적으로 사용 가능

### lock 연산

- 스레드가 임계구역에 들어갈 때 실행되는 **entry** 코드
- 스레드가 잠금 상태(locked)인 락에 접근하게 되면, 블록 상태로 **대기 큐**에 삽입되어 잠들게 된다.
- 잠금 상태가 아니라면 현재 스레드가 락을 잠그고 임계구역으로 진입

### unlock 연산

- 스레드가 임계구역을 나올 때 실행되는 **exit** 코드
- 락을 열린 상태로 변경하고, 대기 큐에 있는 스레드 중 하나를 깨워 **준비 상태**로 변경

### 특징

- 임계구역의 실행 시간이 짧은 경우 **비효율적**
    - 락이 잠겨있는 경우 스레드는 대기 큐에서 기다린다. (**Running** → **Ready**)
    - 대기 큐에 있는 스레드가 락이 풀릴 경우 다시 CPU를 선점한다. (**Ready** → **Running**)
    - 두 번의 **컨텍스트 스위칭**이 일어나 시간 낭비가 될 수 있다.

## 스핀락(Spinlock)

> 뮤텍스와 같은 락을 기반으로 하는 동기화 기법이나, 뮤텍스와 달리 **대기 큐**가 없다. = 공격적인 뮤텍스 (Aggressive Mutex)
> 

### 락 변수

- 스핀락 기법에서의 락 변수는 스핀락이라고도 부른다.
- 스핀락을 **소유**한 오직 한 스레드만이 해당 공유 자원을 독점적으로 사용 가능

### lock 연산

- 스레드가 임계구역에 들어갈 때 실행되는 **entry** 코드
- 스레드가 잠금 상태(locked)인 락에 접근하게 되면, 락이 열릴 때까지 **무한 루프** 진행
- 잠금 상태가 아니라면 현재 스레드가 락을 잠그고 임계구역으로 진입

### unlock 연산

- 스레드가 임계구역을 나올 때 실행되는 **exit** 코드
- 락을 열린 상태로 변경하고, 대기 큐에 있는 스레드 중 하나를 깨워 **준비 상태**로 변경

### 특징

- 뮤텍스의 **non-blocking** 모형
    - 락이 잠겨있을 때 블록 되지 않고 락이 풀릴 때까지 락 변수를 검사하는 코드를 실행하기 때문 : **무한 루프**
- 단일 CPU를 가진 운영체제에서 **비효율적**
    - 락 변수를 검사하는 코드를 실행할 때도 CPU를 사용해야 하기 때문에 CPU의 **낭비**가 심하고, 다른 스레드의 **실행 기회**마저 뺏게 된다.
- **다중 CPU**를 가진 운영체제에서 **서로 다른 CPU** 상에서 멀티스레드가 실행될 때 주로 사용
- 임계구역의 실행 시간이 짧아 빨리 락이 풀리는 경우 **매우 효율적** : 스핀락의 의도
    - 컨텍스트 스위칭과 스케줄링 **불필요**

|  | **뮤텍스** | **스핀락** |
| --- | --- | --- |
| 대기 큐 | 있음 | 없음 |
| block 가능 여부 | 락이 잠겨 있으면 블록(**blocking**) | 락이 잠겨 있어도 블록되지 않고,
계속 락 검사 (**non-blocking**) |
| lock / unlock 연산 비용 | 싸다. | CPU를 계속 사용하므로 비싸다. |
| 하드웨어 | 단일 CPU에 적합 | 멀티코어 CPU에 적합 |
| 주 사용처 | 사용자 모드,
사용자 응용 프로그램 | 커널 모드, 커널 코드,
인터럽트 서비스 루틴 |

## 세마포어(Semaphore)

> 스레드가 동시에 사용할 수 있는 하나의 자원에 대해 스레드가 **공유**하도록 관리하는 기법이자 프로그래밍 개념(concept)
> 

<aside>
💡 **뮤텍스**는 임계구역의 **상호 배제**를  위한 목적이지만, **세마포어**는 자원에 대한 **다중 스레드**의 원활한 **공유**가 목적이다. 즉, 세마포어 값이 2 이상일 경우 **상호 배제**가 되지 않는다는 것.

</aside>

### P 연산과 V 연산

- **P(wait) 연산** : 스레드의 자원 사용의 **허가** 과정
- **V(signal) 연산** : 스레드의 자원 사용이 **끝났음**을 알리는 과정
- **counter 변수** : 사용 가능한 **자원**의 개수를 나타내는 정수형 전역 변수
    - **음수**인 경우 자원을 기다리는 스레드의 개수를 나타낸다.

### 자원을 기다리는 스레드 상태

- **sleep-wait 세마포어** : 사용 가능한 자원이 생길 때까지 **대기 큐**에서 Ready 상태로 대기하고, 사용 가능 자원이 생기게 되면 자원 사용을 허가 받는 Running 상태로 컨텍스트 스위칭
- **busy-waiting 세마포어** : 사용 가능한 자원이 생길 때까지 **무한 루프**를 돌면서 체크하고, 사용 가능 자원이 생기게 되면 자원 사용을 허가 받는 방식 ⇒ **대기 큐**가 필요 없다.

### 자원의 인스턴스 개수

- **카운터 세마포어 (Counter Semaphore)** : 자원의 인스턴스가 **여러 개**인 경우
- **이진 세마포어 (Binary Semaphore)** : 자원의 인스턴스가 **한 개**인 경우
    - **상호 배제** 관리 기법이므로 **뮤텍스**와 매우 유사
