텍스트: 아키텍처 및 트랜잭션 동시성 제어

- 키워드로 요약하는 구조
    - SGA 자료구조
    
    # DB 버퍼 캐시
    
    - DBA 번호를 얻어 데이터 블록 해싱
        - 같은 DBA 블록 → 해시 버킷에 연결리스트
        - 해시버킷 → 버퍼 헤더(얘가 연결리스트) = **해시 체인** → 버퍼 블록
            - 버퍼 헤더에 있는 포인터로 버퍼 블록 찾아감
    
    # 캐시 버퍼 체인
    
    - **`래치는 SGA 자료구조 내의 진입을 보호하는 목적임`**
    - 래치는 해시 체인을 스캔, 블록 추가, 제거할 때마다 요구
    - 해시 체인은 **`래치`**로 보호
        - 같은 리소스에 대한 액세스 직렬화를 위해 필요
            - 같은 해시 체인 → 진입해서 새로운 버퍼 블록 연결 / 해제 동시에 진행 → 방지
                - **`cache buffers chains` 래치**
                    - 하나의 cache buffers chains 래치가 여러 개 해시 체인을 관리
                    - 목표 = 하나의 체인에 하나의 버퍼만 달리는 것
                        - 이러려면 해시 버킷 개수가 많아야 함 → 실제로 많다고 함
                            - 근데 해시 버킷이 많다고 해서 dba 블록의 충돌까지 막을 수 있나. 이건 해시값인데. 해시 알고리즘이 해시 버킷이 많으면 알아서 분산을 해주는 쪽으로 되어있나 보다.
    
    # 캐시 버퍼 LRU 체인
    
    - 버퍼 헤더는 LRU 체인에도 연결되어 있다
        - 메모리 유한성으로 인해 자주 안쓰이는거 뒤로 미루다가 밀어낸다는거임
    - 내부적으로 두 개
        - Dirty 리스트
            - 변경되었지만 아직 디스크에 기록되지 않은 Dirty 버퍼
        - LRU 리스트
            - Dirty 리스트로 옮겨지지 않은 나머지 관리
    - 모든 버퍼 블록은 두 개의 리스트 중 하나에 속함
    - 액세스되는 동안에는 잠시 리스트에서 빠져나왔다가 변경 여부에 따라 둘 중 하나로 다시 복귀
    
    ### 버퍼 상태 (캐시 버퍼의 상태를 의미함)
    
    - Free
        - 비어있거나
        - 데이터가 이미 동기화된 상태여서 언제든지 덮어써도 무방한 상태
    - Dirty
        - 변경 → 아직 동기화 x인 데이터들
        - 디스크에 기록 → Free로 바뀜
    - Pinned
        - 현재 액세스 중
    
    # 버퍼 Lock
    
    - 버퍼 블록 찾음 → 래치 빨리 놔버려야 함
        - 래치 경합 발생 가능성이 높아져서
    - 그래서 래치를 해제했다고 하자
        - 근데 선행 프로세스가 아직 버퍼를 사용중이면?
        - 문제 = 두 개 이상의 프로세스가 동시에 버퍼의 내용을 쓸 수 있다 (래치를 놓았는데 누가 접근에 성공해버렸다는 것) → **`정합성`** 문제
    - 이거 방지하려고 버퍼 헤더로부터 버퍼 Lock 받는다

    - 이제야 안심하고 버퍼 블록 읽고 씀
        - Share / Exclusive에 따라 액세스 직렬화 여부 판단
    - 다른 프로세스가 래치 획득, 하지만 블록의 버퍼 Lock이 Exclusive모드인 경우
        - 래치 놓는다
        - 버퍼 Lock 대기자 목록에 등록
            - buffer busy waits 대기 이벤트가 이 순간에 나타남
    - 버퍼 Lock 가진 프로세스가 작업 끝
        - 다시 래치 획득함
        - 이 짧은 순간에 버퍼 헤더를 액세스하려는 다른 프로세스와 충돌이 생길 수 있어서
            - 이 때 래치 획득 우선순위는 버퍼 Lock을 점유한 녀석이 가져간다는 의미인 듯
    - 버퍼 Lock을 해제하고 래치를 해제해야 버퍼 블록 읽기 작업이 완료되는 것
    
    # 버퍼 핸들
    
    - 버퍼 Lock = 버퍼 Pin
        - 사용중이라고 표시해둔 것
    - 버퍼 Pin은 읽기 → 여러 명, 쓰기 → 1명 가능
    - 버퍼 Pin을 설정하기 위해 사용하는 오브젝트가 버퍼 핸들
    - 버퍼 핸들 얻어서 버퍼 헤더에 있는 소유자 목록에 연결시키는 방식으로 Pin 설정
    - 버퍼 핸들에도 래치 필요
        - cache buffer handles 래치
        - 여기도 동시에 많이 접근하면 경합지점
            - 따라서 오라클이 기본적으로 5개 주고 더 필요하면 할당한다고 한다.
    
    # 버퍼 Lock 필요성
    
    - DML Lock 외에도 왜 블록에 또 다른 Lock을 담냐
        - 블록 단위 I/O라서 그렇다
    - 로우 Lock 동시 설정
    - 블록 SCN 변경, ITL 슬롯 변경 등..
    - 위의 것들을 방지하지 않으면 Lost Update 발생 → **`정합성`** 문제
    - 그래서 **`블록 자체로의 진입을 직렬화`**한다
    
    # 버퍼 Pinning
    
    - 바로 버퍼 Pin 해제 안하고 데이터베이스 Call 내에 유지
        - 같은 블록을 재방문할 가능성이 큰 몇몇 오퍼레이션을 수행할 때만
            - 옵티마이저는 도대체 누가 설계했길래 이걸 어떻게 알고 이런 미친 짓이 가능하냐
        - 래치 획득 과정 없이 간다면 buffer is pinned count 항목의 수치가 증가
    - 버퍼 Pinning을 통해 블록 I/O 감소 효과가 있다
        - **`그래서 매우 중요하다`**
        - 인덱스 루트 블록 / 브랜치 블록 Pinning..이 그 예시
            - 이 말은 뭐다? CF가 좋으면 이런 효과도 좋다는 것
    
    # Undo 세그먼트 트랜잭션 테이블 슬롯
    
    - Undo 세그먼트 헤더
        - 트랜잭션 테이블 슬롯이 위치함
            - 트랜잭션 ID
            - 트랜잭션 상태정보
            - 커밋 SCN
            - Last UBA (Undo Block Address)
            - 기타
    - 순서
        - 트랜잭션 시작 → Undo 세그먼트의 트랜잭션 테이블 슬롯 할당 → Active 상태 표시
            - 바로 못얻으면 **`tx undo segment tx slot`** 이벤트
        - 데이터 / 인덱스 블록 변경 → Undo 블록에 Undo 레코드 기록
            - Insert에 추가된 레코드 rowid
            - Update에 변경된 레코드의 이전 이미지
            - Delete에 삭제된 컬럼의 이미지
        - Last UBA에는 트랜잭션 기록사항을 거꾸로 기록함
            - Undo 레코드에 체인 형태로 달아서 관리하고 롤백할 때 거꾸로 처리
        - Active 상태의 트랜잭션 블록 = 다른 트랜잭션이 못씀
            - 커밋되면 Committed 기록
            - 이후 커밋 SCN 기록
        - 가장 먼저 커밋된 트랜잭션 슬롯부터 순차적으로 재사용 → Undo 데이터는 오래 남아있음
    
    # 블록 헤더 ITL 슬롯
    
    - 데이터 블록과 인덱스 블록 헤더에 있음
        - ITL 슬롯 번호
        - 트랜잭션 ID
        - UBA
        - 커밋 Flag
        - Locking 정보
        - 커밋 SCN
    - 아래 블록 dump에 위의 정보가 나타나는 것
        
        ```sql
        It1   Xid                   Uba    Flag    Lck      Scn/Fsc
        0x01  0x0004.00c.000035c3   ~~     --U-      1      fsc 0x~
        0x02  0xffff.000.00000000   ~~     C---      0      scn 0x~
        0x03  0x0005.00d.000003c4   ~~     C-U-      0      scn 0x~
        ```
        
    - 순서
        - 특정 블록에 속한 레코드 갱신을 → 블록 헤더로부터 ITL 슬롯 획득
            - 할당 못받으면 트랜잭션 블로킹
            - **`enq : TX - allocate ITL enty`** 대기 이벤트
            - 이거 막으려고 옵션 제공하는데 initrans, maxtrans, pctfree 등
        - 트랜잭션 ID 기록 후 **`Active`**로 변경
    
    # Lock Byte
    
    - 로우마다 헤더에 Lock Byte → 해당 로우 갱신중인 트랜잭션 ITL 슬롯 번호 기록
    - 레코드에 Lock Byte 있음 → ITL 슬롯 찾아감 → active 상태 풀릴 때까지 대기
    
- 오라클은 **데이터베이스`(데이터를 저장하는 파일집합)`**과 이를 액세스하는 프로세스 사이에 **`SGA`**라는 메모리 캐시 영역을 둔다
- 프로세스가 동시에 데이터에 액세스
    - 사용자 데이터를 보호하는 **`Lock`**과 공유 메모리 영역인 SGA상에 위치한 데이터 구조에 대한 액세스를 직렬화하기 위한 **`래치`** 필요
- 백그라운드에서 DBWR과 CKPT 프로세스가 캐시와 데이터파일 간 **`동기화`**를 주기적으로 수행
- **디스크에 저장된 집합**(Datafile, Redo Log File, Control File 등)을 **`데이터베이스`**
- **SGA 공유 메모리 영역과 이를 액세스하는 프로세스 집합**을 합쳐서 **`인스턴스`**
    - 프로세스 집합은 다시 서버 프로세스와 백그라운드 프로세스 집합으로
        - 서버 프로세스는 일종의 포그라운드 프로세스

# 오라클 접속 과정

- 각 클라이언트를 위한 전용 서버 프로세스가 뜬다(Shared Server가 아닌 경우)
- SQL 파싱 → 최적화 → 커서 열고 SQL 수행 → 블록 읽음 → 결과 네트워크 전송
    - 이거 서버 프로세스가 모두 처리한다.
    - 도중에 스스로 처리 못하는 것들(DB 파일 버퍼 캐시 적재, Free 블록 확보, Redo 로그 버퍼 비우기 등..)은 **OS나 I/O, 백그라운드 프로세스 등에 신호**를 보낸다.

### 순서

- 클라이언트가 LISTNER에 연결 요청 → SERVER에 프로세스 생성 요청 → PGA 생성 → 클라이언트에게 RESEND 패킷 전송 → 연결
    - 리스너에 연결 요청을 하는 순간 하나의 프로세스 생성 및 PGA 생성되는 점 염두
    - 따라서 **재사용성** 염두 → **`커넥션 풀`**

### RAC

- 하나의 데이터베이스를 액세스하는 다중 인스턴스로 구성
    - 오라클은 **`글로벌 캐시`** 제공
        - 로컬 캐시에 없는 데이터 블록 이웃한테 전송해주는게 가능
    - Active 상태의 블록 (다른 인스턴스에서 갱신하고 아직 커밋하지 않은 것)까지, 즉 Drity 버퍼 상태에서 블록을 주고받을 수도 있다

# DB 버퍼 캐시

- SGA **구성 요소 중 하나**
    - 내가 헷갈렸던 것 = SGA를 DB 버퍼캐시와 동일하다고 생각한 것
- 모든 처리 단위는 **`블록`**

### 구조

- **해시 테이블** 구조로 관리
- 데이터 블록을 해싱하기 위해 사용되는 키 값은 **`DBA`**
    - 해시 함수에 DBA 입력 → 해시 값 리턴 → 같은 블록들은 같은 **`해시 버킷`**에 **`연결리스트`**
        - 연결리스트가 **`해시 체인`**임
    - 찾고자 하는 데이터 블록 주소 해시값 변환 → 해시 버킷에서 체인 따라가 스캔 → 없으면 디스크 접근한 후 **`해시 체인에 연결`**한다
- 실제로는 **해시 버킷 → 버퍼 헤더 → 버퍼 블록**
    - 헤더에 있는 포인터를 이용해서 다시 버퍼 블록 찾아가는 것

### 캐시 버퍼 체인

- 각 해시 체인은 **`래치`**로 보호
    - 래치를 획득한 프로세스만 진입 가능
        - 이 래치가 **`cache buffers chains`** 래치
            - 버퍼 헤더에 pin을 걸어두기도 함
        - 여러 개 해시 체인을 동시에 관리함
- 하나의 체인에 하나의 버퍼만 달리는 것을 목표로 해야한다
    - 그러려면 해시 버킷 개수가 충분히 많아야 한다.
- 래치는 데이터 자체를 보호하는 것이 아니라 SGA에 공유되어 있는 자료구조를 보호하는 것임

### 캐시 버퍼 LRU 체인

- 모든 버퍼 블록 헤더를 LRU 체인에 연결해 사용빈도 순으로 옮긴다
    - 액세스 빈도에 따라 데이터 블록들을 밀어낸다
- 2가지 존재
    - Dirty 리스트
    - LRU 리스트
        - LRU 리스트를 보호하기 위해 사용하는 래치가 cache buffers lru chain
    - 모든 버퍼블록은 이 둘 중에 하나에 반드시 속한다
        
        

# 버퍼 Lock

- 버퍼 블록을 찾았으면 가급적 래치를 빨리 해제해야 함
    - 그렇지 않으면 경합 가능성 多
- 데이터 정합성을 위해
    - 캐시된 버퍼 블록을 읽거나 변경하려는 프로세서는 버퍼 헤더로부터 **`버퍼 Lock`**을 획득한다
    - 버퍼 Lock을 획득하면 바로 래치를 해제한다
        - 일종의 안전장치
- 버퍼 Lock 읽기만 할 때는 Share 모드, 변경 때는 Exclusive
- 체인 래치를 획득하고 버퍼를 찾았는데 다른 프로세스가 버퍼 Lock을 가지고 있는 경우
    - 대기자 목록에 기록하고 래치는 해제한다
        - 이 때 발생하는 이벤트가 **`buffer busy waits`**
- 작업을 완료하면 버퍼 Lock을 해제해야 한다
    - 이 때 해제를 위한 체인 래치를 다시 획득함
    - 이후 버퍼 Lock을 해제하고 래치를 해제해야 버퍼 블록 읽기가 완료된다.

### 버퍼 핸들

- 버퍼 Lock을 설정하는 것은 현재 그 버퍼를 사용하고 있다는 소리임
    - **`버퍼 Pin`**이라고 하기도 한다
        - 변경 시에는 하나의 프로세스만 Pin 설정 가능
        - 읽기 시에는 여러개
- 버퍼 헤더에 Pin을 설정하려고 사용하는 오브젝트가 버퍼 핸들
    - 버퍼 핸들을 얻어 버퍼 헤더에 있는 소유자 목록에 연결시키는 방식으로 Pin 설정
        - 버퍼 핸들 얻기 위한 래치 이벤트 = **`cache buffer handles`**
    - 이 경합을 피하기 위해 _*db*_handles_cached 개수만큼의 버퍼 핸들을 미리 할당함
        - 부족할 때만 래치 얻는다

### 버퍼 Lock 필요성

- 데이터 변경 시 DML Lock 외에 또 다른 Lock을 획득하는 이유가 무엇인가
    - 블록 단위로 I/O를 수행하기 때문에
- 블록 자체로의 진입을 직렬화해야 함

### 버퍼 Pinning

- 버퍼 Pin을 데이터베이스 Call이 진행되는 동안 유지함
    - 논리적 블록 읽기 횟수 줄이기에 효과적
    - 하나의 데이터베이스 Call 내에서만 유효

# Redo

- 데이터 파일과 컨트롤 파일에 가해지는 모든 변경사항을 하나의 Redo 로그 엔트리로서 Redo 로그에 기록한다
- Online Redo 로그
    - Redo 로그 버퍼에 버퍼링된 로그 엔트리를 기록하는 파일
        - 최소 2개 이상으로 구성
        - 용량에 따른 로그 스위칭 발생하며, round-robin으로 재사용한다
- Archived Redo 로그
    - Online Redo 로그가 재사용되기 전에 다른 위치로 백업해둔 파일

### 사용 용도

- Database Recovery
    - 디스크 깨짐 등의 발생 시 Archived Redo 로그를 이용해 복구
- Cache Recovery
    - roll forward = Online Redo 로그 이용해서 사고 발생 직전까지 수행했던 트랜잭션들 재현
    - roll backward = Transaction Recovery 진행, 커밋되지 않았던 트랜잭션 모두 롤백
    - 데이터 파일에는 커밋에 성공한 데이터만 남게 된다.
- Fast Commit
    - 변경된 메모리 버퍼 블록을 디스크 상의 데이터 블록에 기록하는 작업은 Random Access
    - 로그는 Append 방식으로 기록
        - 변경사항을 Append 방식으로 빠르게 로그 파일에 기록한다
        - 이후 DBWR을 이용해 나중에 배치 방식으로 일괄 수행
    - 커밋 정보가 로그에만 기록돼 있기만 한다면 안심 커밋 가능
