## 선분이력 기본 조회 패턴

```sql
select 개월수, 금액
from 은행
where 고객번호 = '123'
and :dt between 시작일 and 종료일
```

- 2024년 10월 10일 데이터를 조회하려면 dt에 '20241010'을 입력하면 된다.

```sql
select 개월수, 금액
from 은행
where 고객번호='123'
and 종료일='99991231'
```

- 현재 시점을 조회할 때는 종료일 = '99991231' 이런 식으로 입력하여 처리가능

### 인덱스가 [시작일 + 종료일] 구성일 때 최근 시점 조회

- 특정 고객에 대한 최근 시점 정보를 조회하는 경우
- :dt에 '20101010'을 넣었다고 하자

```sql
select *
from 은행
where 고객번호='123'
and '20101010' between 시작일 and 종료일

# between을 풀면 아래와 같음

where 고객번호='123
and 시작일<='20101010'
and 종료일>='20101010'
```

- 시작일 컬럼이 '<=' 조건이므로 20101010 이전 이력 레코드를 모두 스캔한다.
- 종료일 컬럼 역시 '>=' 조건이므로 20101010 이후 이력 레코드를 모두 스캔한다.
- 선두 컬럼인 '시작일' 조건이 부등호이기 때문에 종료일 조건은 **스캔 범위를 줄이는데 기여하지는 못한다**.

(그림을 그려보며 판단하면 이해가 빠름)

- 인덱스가 [종료일 + 시작일]인 경우 최근 데이터 조회에 유리하며
- 인덱스가 [시작일 + 종료일]인 경우 오래된 과거 데이터 조회에 유리하다.
- 중간 지점은 둘 다 어느 정도의 비효율을 감수해야한다.
- **이를 모두 효율적으로 사용하는 방법은 rownum <= 1과 index_desc 힌트를 적절히 사용하는 것.**

```sql
select/*+ index_desc(a idx_x01) */ *
from 은행
where 고객번호='123'
and '20101010' betwwen 시작일 and 종료일
and rownum <= 1
```

## Index Skew

- 인덱스 엔트리가 왼쪽 또는 오른쪽에 치우치는 현상
- 예시 : 1 ~ 10만까지 데이터를 삽입하고, 5만까지의 데이터를 삭제한 경우
- 빈 인덱스 블록들은 커밋하는 순간 freelist로 반환되지만 **인덱스 구조 상에는 그대로 남는다.**

**- 커밋하지 않으면 반환되지 않고, 새로운 블록을 요구하게 될 것임**

- 상위 브랜치에서 해당 리프 블록을 가리키는 엔트리가 그대로 남아있기 때문에, 인덱스 정렬 순서상 입력될 새로운 값이 들어오면 언제든 재사용 가능하다 (freelist에서 바로 제거되지 않고, 빈 블록을 찾는 스캔 과정에서 제거됌)
- 인덱스 분할 시에도 재사용
- **Skew의 문제 : 다시 채워질 때까지 인덱스 효율이 낮다.**

## Index Sparse

- 인덱스 블록 전반에 걸쳐 밀도가 낮음 (책 174 참고)
- 지워진 자리에 새로운 값이 입력되지 않으면 영영 재사용되지 않을 수도 있다.
- 총 레코드 건수가 일정한데도 인덱스 공간 사용량이 계속 커지는 것

## 소트 오퍼레이션을 생략하기 위한 컬럼

```sql
create index t_idx on t(a, b, c, d)

select *
from t
where a = 1 and b = 1
order by a, c, b, d;
```

- **인덱스 구성 컬럼이 조건절에서 '=' 연산자로 비교된다면,** 그 컬럼은 order by 절에서 누락되거나 인덱스와 다른 순서로 기술하더라도 상관없다. group by에도 동일하게 적용.
- 일부 불가능 예시

```sql
select *
from t
where a = 1
order by c

select *
from t
where a = 1 and b between 1 and 2
order by c, d

select *
from t
where a = 1 and b between 1 and 2
order by a, c, b
```

## 스칼라 서브쿼리 캐싱 효과

스칼라 서브쿼리의 입력 값은 거기서 참조하는 메인 쿼리의 컬럼 값을 의미

```sql
select a, b,
	(
    	select d.a      # 출력값 : d.a
        from dept d
        where d.a = e.a # 입력값 : e.a
    )
from emp e
where sal >= 2000
```

- 해시 충돌이 발생한 입력값이 반복적으로 입력되면 스칼라 서브쿼리도 반복 수행
- 입력 값의 종류가 소수여서 해시 충돌 가능성이 적은 때에 효과가 있다.
