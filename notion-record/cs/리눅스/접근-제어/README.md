# 접근 제어

### 임의 접근 제어 (direction access control)

- 사용자의 신분을 기반으로 리소스에 대한 접근을 제한한다.
- **`임의는 특정 권한을 가진 사용자가 이를 다른 사용자에게 전달할 수 있다는 것`**.
    - 루트 유저가 아니어도 권한 부여 = 접근 가능(rwx).

### 강제 접근 제어

- 반드시 관리자가 모든 권한 부여.

### 시스템 사용자 or 시스템 계정(미완성)

- 일반적으로 프로그램(daemon)은 이런 유형의 계정을 사용하여 백그라운드 프로세스 실행. 필요에 의해 자동으로 생성되는 계정도 있고, 시스템 계정을 만들 수도 있다.
    - bin - 명령어 관리 계정
    - daemon - 시스템 데몬 계정
    - adm - 시스템 로그 관련 계정
    - 일반적으로 로그인 불가능(일반적으로.. 아닌경우는 뭐지)

### UID 65534, **`nobody 사용자` 에 대해**

- https://www.baeldung.com/linux/nobody-user
- nobody 유저는 최소한의 권한만 가지고 있다. 보안을 중요시 할 때 사용함.
네트워크를 통해 들어온 불특정 다수가 권한이 많으면 문제가 된다. 따라서 nobody 사용자의 권한을 부여해 서버 리소스 접근을 최소화한다.

### SUID와 chmod

- suid를 쓸 수 있는 상황이 있겠으나 어떤 상황인지.
- 일반 사용자에게 제한된 작업을 수행할 수 있도록 허용하면서도 보안을 유지
    - 실행 권한을 주되 모든 상황에서 해당 권한을 주고 싶지는 않을 때(특별한 상황에서만 부여한다)
- **`SUID`**는 `chmod` 명령으로 설정할 수 있는 **특수 비트**로, 프로그램 실행 시 그 프로그램의 소유자 권한을 임시로 부여합니다. 일반적으로, 실행하는 사용자의 권한으로 프로그램이 실행되지만, `SUID`가 설정된 경우 소유자의 권한으로 실행됩니다.

### **`SUID`의 주요 목적과 장점**

- **특정 시스템 작업 수행**: 특정 작업은 보통 사용자 권한으로는 수행할 수 없고, 시스템 관리자(보통 `root`) 권한이 필요합니다. 예를 들어, 비밀번호 변경 명령어(`/usr/bin/passwd`)는 일반 사용자가 자신의 비밀번호를 변경할 수 있도록 `SUID`가 설정되어 있습니다. 이렇게 하면 `passwd` 프로그램이 `root` 권한을 사용하여 `/etc/shadow` 파일을 수정할 수 있습니다.
- **제한된 권한 상승**: `SUID`를 통해 프로그램이 필요할 때만 특정 권한을 사용할 수 있도록 하여, 보안성을 유지하면서도 작업을 수행할 수 있습니다. 반대로, 모든 사용자에게 권한을 부여하는 경우에는 보안 문제가 발생할 수 있습니다.
- **보다 세밀한 접근 제어**: 모든 사용자에게 특정 파일이나 시스템 리소스에 대한 `chmod`로 접근 권한을 부여하는 것은 보안 위험을 초래할 수 있습니다. `SUID`를 사용하면 특정 프로그램이 실행될 때에만 일시적으로 권한이 상승하므로 더 안전한 방법이 됩니다.
