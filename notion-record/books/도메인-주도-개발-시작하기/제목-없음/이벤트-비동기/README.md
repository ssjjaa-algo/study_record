# 이벤트 - 비동기

이벤트간 신경써야 하는 요소는 외부 시스템 성능 저하로 인한 영향, 트랜잭션 처리 등이 있다. 외부 시스템과의 연동을 동기로 처리할 때 발생하는 문제들의 해결 방법으로 비동기 처리, 이벤트와 트랜잭션 연계 등이 있다.

# 비동기 이벤트 처리

회원가입 신청을 하자마자 이메일 인증을 위해 바로 내 메일함에 이메일이 올 필요는 없다. 만약 받지 못한다면, 다시 보내는 기능을 이용하면 된다. 서비스에 따라 주문을 취소하자마자, 바로 결제를 취소하지 않아도 된다. 

‘A하면 이어서 B하라’ → ‘A하면 최대 언제까지 B하라’

여기서 A하면이 이벤트이며, 이벤트를 비동기로 처리할 수 있는 지점으로 해석할 수 있다. 다시 말해, 이벤트가 발생하면 별도 스레드로 B를 수행하는 핸들러를 실행하는 방식을 의미한다.

## 로컬 핸들러 비동기 실행

이벤트 핸들러를 별도 스레드로 실행하자. Spring에서는 @Async, 아래 두 가지 필요

- Application에 @EnableAsync 활성화
- 이벤트 핸들러 메서드에  @Async

## 메시징 시스템을 이용한 비동기 구현

카프카, 래빗MQ와 같은 메시징 시스템을 이용하는 것. 이벤트 발생 시 이벤트 디스패처가 이벤트를 메시지 큐에 보내며, 메시지 큐가 이벤트를 메시지 리스너에 전달하고 리스너가 알맞은 이벤트 핸들러를 이용해서 이벤트를 처리한다. 이 때, 이벤트를 메시지 큐에 저장하는 과정과 메시지 큐에서 이벤트를 읽어와 처리하는 과정은 별도 스레드나 프로세스로 처리된다.

<aside>
💡 Order → Events → 메시징 시스템 → MessageListner → 이벤트핸들러

</aside>

메시지 큐를 사용하면 보통 이벤트를 발생시키는 주체와 이벤트 핸들러가 별도 프로세스에서 동작한다. 이 것의 의미는, 이벤트 발생 JVM과 이벤트 처리 JVM이 다르다는 것을 의미한다. 

## 이벤트 저장소 이용

일단 DB에 이벤트를 저장하고, 별도 프로그램을 이용해서 이벤트 핸들러에 전달하는 방법.

<aside>
💡 도메인 → 이벤트 디스패처 → 로컬 핸들러 → DB → 포워더 → 이벤트 핸들러

</aside>

포워더는 주기적으로 이벤트 저장소를 체크하는 역할이다. 포워더를, API로 대체할 수 있다.
