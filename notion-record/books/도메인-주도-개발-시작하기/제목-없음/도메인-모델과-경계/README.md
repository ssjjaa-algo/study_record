# 도메인 모델과 경계

모델은 특정한 컨텍스트 하에서 완전한 의미를 가질 수 있다. 같은 제품이라고, 카탈로그 컨텍스트와 재고 컨텍스트에서 의미가 서로 다르다. 하여, 구분되는 경계를 가지는 컨텍스트가 필요한데, DDD에서는 **바운디드 컨텍스트**(Bounded Context)라고 부른다.

## 바운디드 컨텍스트

논리적으로 한 개의 모델을 가지며, 용어를 기준으로 분리한다. 한 개의 바운디드 컨텍스트가 여러 하위 도메인을 포함하더라도, 하위 도메인마다 구분되는 패키지를 가지도록 구현하는게 좋다.

## 구현

도메인 기능을 사용자에게 제공하는데 필요한 표현, 응용 서비스, 인프라스트럭처 영역을 모두 포함한다. 도메인 모델의 데이터 구조가 바뀌면, DB 테이블 스키마에 영향이 가므로 테이블도 바운디드 컨텍스트에 포함된다.

## 바운디드 컨텍스트 간 통합

카탈로그 하위 도메인에 추천 기능을 추가한다고 하면, 카탈로그 바운디드 컨텍스트와 추천 바운디드 컨텍스트가 생긴다. 서로 다른 데이터 포맷을 사용한다면, 카탈로그는 추천을 통해 받은 데이터를 변환해야 한다. 복잡하면 translator 따로 두어서 사용해도 된다.
이 때, REST API를 사용하는 것은 두 바운디드 컨텍스트를 직접 통합하는 방식이며 간접 통합 방식으로는 메시지 큐를 이용하는 방법이 있다.

메시지 큐를 이용하는 경우, 두 바운디드 컨텍스트는 메시지의 데이터 구조를 맞춰야 한다. 카탈로그 관점, 추천 관점에서의 모델은 서로 다른 데이터 모습을 보이며 이 지점에서 팀 간의 회의가 필요하다.

## 바운디드 컨텍스트 간 관계

가장 흔한 관계는 한 쪽이 API 제공 / 한 쪽이 API 호출하는 그림으로, REST API가 대표적이다. 카탈로그 바운디드 컨텍스트가 추천 상품을 보여주기 위해, 추천 바운디드 컨텍스트를 이용한담면 카탈로그가 하류가 되며, 추천이 상류가 된다. 즉, 추천이 제공하는 데이터와 기능에 의존하게 되는 모습이다.

상류 컴포넌트는 일종의 공급자이며, 하류 컴포넌트는 고객이다. 하여, 상류와 하류는 변경에 영향을 미치기 때문에 공유 / 일정 협의가 중요하다.

두 바운디드 컨텍스트가 같은 모델을 공유하는 경우도 존재한다. 이 경우, 중복 설계를 막을 수 있는 지점이며 두 팀이 공유하는 모델을 **공유 커널**이라고 한다. 공유하기 떄문에 발생하는 장단은, 중복 제거와 임의 변경간 문제 발생 가능성이다.

## 컨텍스트 맵

바운디드 컨텍스트에 매몰되지 말고, 일종의 마인드맵 형식 (OHS, ACL) 정도 표시하여 표현

시스템의 전체 구조를 보여주며, 별도로 그리는 규칙은 따로 없다.
