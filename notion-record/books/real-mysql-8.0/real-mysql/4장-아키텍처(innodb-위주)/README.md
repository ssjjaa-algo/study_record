# 4장 아키텍처(InnoDB 위주)

MySQL 서버는 크게 MySQL 엔진과 스토리지 엔진으로 구분할 수 있다. 아래에서 기본 개념들을 정리한다.

1. MySQL 엔진

클라이언트로부터 접속 및 쿼리 요청을 처리하는 커넥션 핸들러, SQL 파서 및 전처리기, 옵티마이저가 중심을 이룬다.

1. 스토리지 엔진

 MySQL 엔진이 SQL 문장을 분석하고, 최적화 하는 작업을 처리하고 나서 스토리지 엔진은 실제 데이터를 디스크에 저장하거나, 디스크로부터 읽어들이는 부분을 전담한다.

1. 핸들러 API

MySQL 엔진에서 분석, 최적화한 결과를 스토리지 엔진에 넘겨주면 스토리지 엔진에서 데이터를 읽거나 저장한다고 했다. 즉 MySQL 엔진이 스토리지 엔진에 요청하는 것을 핸들러(Handler)라고 한다. 최소한 MySQL 엔진이 스토리지 엔진에게 데이터를 읽어오거나 저장하도록 명령하려면 **반드시 핸들러를 통한다**는 점은 기억해두자.

## MySQL 스레딩 구조

프로세스 기반이 아니라 스레드 기반으로 작동하며, **포그라운드**와 **백그라운드** 스레드로 구분할 수 있다.

(참고) 서버에서 실행중인 스레드 목록은 performance_schema의 threads 테이블을 통해 확인할 수 있다.

1. 포그라운드 스레드

최소 MySQL에 접속한 클라이언트의 수만큼 존재하며, 각 클라이언트 사용자가 요청하는 쿼리 문장을 처리한다. 클라이언트가 작업을 마치고 커넥션을 종료하면 해당 커넥션을 담당하는 스레드는 스레드 캐시로 되돌아가며, 이미 캐시에 일정 개수 이상의 스레드가 대기중이면 캐시에 들어가지 않고 스레드를 종료하게 된다.

포그라운드 스레드는 데이터를 데이터 버퍼나 캐시로부터 가져오며, 버퍼와 캐시에도 없는 경우에는 디스크나 인덱스 파일을 통해 읽어와서 작업을 처리한다. 

1. 백그라운드 스레드

쓰기 스레드는 많은 작업을 백그라운드로 처리한다. 사용자의 요청을 처리하는 것 중 쓰기 작업은 지연(버퍼링) 되어 처리할 수 있지만 읽기 작업은 절대 그럴 수 없다. InnoDB에서는 Insert, Update, Delete 쿼리의 경우 데이터가 디스크의 데이터 파일로 완전히 저장될 때 까지 기다리지 않아도 된다.

## 메모리 할당 및 사용 구조

1. 글로벌 메모리 영역

클라이언트 스레드의 수와 무관하게 하나의 메모리 공간만 할당되며, 필요에 따라 2개 이상의 메모리 공간을 할당받을 수 있다. 생성된 글로벌 영역은 N개라 하더라도 모든 스레드의 의해 공유되며, 대표적인 영역은 아래와 같다.

1. 테이블 캐시
2. InnoDB 버퍼 풀
3. InnoDB 어댑티브 해시 인덱스
4. InnoDB 리두 로그 버퍼

1. 로컬 메모리 영역

세션 메모리 영역이라고도 하며, 클라이언트 스레드가 쿼리를 처리하는데 사용하는 영역이다. 클라이언트 스레드가 사용하는 메모리 공간이라고 해서 클라이언트 메모리 영역이라고도 한다. 로컬 메모리 영역이라는 이름 답게 클라이언트 스레드 별로 독립적으로 할당되며 절대 공유되지 않는다. 대표적인 영역은 아래와 같다.

1. 정렬 버퍼
2. 조인 버퍼
3. 바이너리 로그 캐시
4. 네트워크 버퍼

## 쿼리 실행 구조

1. 쿼리 파서

사용자 요청으로 들어온 쿼리 문장을 토큰으로 분리해 트리 형태로 만든다. 기본 문법 오류 등이 이 과정에서 발견된다.

1. 전처리기

쿼리 파서를 통해 만들어진 트리를 기반으로 쿼리 구조에 구조적인 문제점이 있는지 확인한다. 해당 객체의 존재 여부와, 객체의 접근 권한 등을 확인하여 걸러내는 작업을 한다. 가령 존재하지 않다거나, 권한 이슈로 사용하지 못하는 경우를 말한다.

1. 옵티마이저

최적화를 담당하는 DBMS의 두뇌다.

## InnoDB 스토리지 엔진 아키텍처

### 1. 프라이머리 키에 의한 클러스터링

InnoDB의 모든 테이블은 기본적으로 PK를 기준으로 클러스터링되어 저장된다. PK 값의 순서대로 디스크에 저장된다는 뜻이며, 모든 세컨더리 인덱스는 레코드의 물리 주소 대신  PK값을 논리적인 주소로 사용한다. PK는 클러스터링 인덱스 자체이기 때문에 PK를 이용한 레인지 스캔은 상당히 빨리 처리될 수 있으며 쿼리 실행 계획에서 다른 인덱스에 비해 높은 확률로 선택된다.

### 2. MVCC (Multi Version Concurrency Control)

MVCC의 가장 큰 목적은 **잠금을 사용하지 않는 일관된 읽기**를 제공하는데 있다. InnoDB에서는 언두 로그를 통해 기능을 구현하며, 멀티 버전이라 하는 것은 하나의 레코드에 대해 여러 개의 버전이 동시에 관리된다는 뜻이다.

격리 수준이 READ UNCOMMITED인 경우에서는 데이터가 변경된 것을 바로 읽어들일 수밖에 없다. 하지만 READ COMMITED이상의 격리 수준에서는 커밋되지 않은 경우 변경된 내용 대신 이전의 내용을 보관하고 있는 언두 영역의 데이터를 반환한다. 즉 하나의 레코드에 대해 여러 개의 버전이 유지되며 필요에 따라 어느 데이터가 보여지는 지는 여러 상황에 따라 달라지는 구조이다.

UPDATE 쿼리가 실행되면 InnoDB 버퍼 풀은 즉시 새로운 데이터로 변경되고 기존 데이터는 언두 영역으로 복사되는데, COMMIT 명령을 실행하면 더 이상의 변경작업 없이 지금의 상태를 영구적인 데이터로 만든다. 롤백의 경우, 언두 영역의 데이터를 InnoDB 버퍼 풀로 다시 복구한다.

### 2-1. 잠금 없는 일관된 읽기

InnoDB 스토리지 엔진은 MVCC 기술을 이용해 잠금을 걸지 않고 읽기 작업을 수행한다. 격리 수준이 SERIALIZABLE 아래에 한해서다. SELECT 같은 작업은 다른 트랜잭션의 변경 작업과 관계없이 바로 실행된다.이를 잠근 없는 일관된 읽기라고 표현한다.

### 3. InnoDB 버퍼 풀

디스크의 데이터 파일이나 인덱스 정보를 메모리에 캐시해두는 공간이다. 쓰기 작업을 일괄 작업으로 처리할 수 있게 해주는 버퍼 역할도 같이 한다. InnoDB 스토리지 엔진에서 가장 핵심적인 부분이다.

### 3-1. 버퍼 풀의 구조

메모리 공간을 페이지 크기의 조각으로 쪼개어 필요로 하는 데이터를 각 조각에 저장한다.

### 4. 언두 로그

트랜잭션과 격리 수준을 보장하기 위해 DML로 변경되기 이전 버전의 데이터를 별도로 백업한다. 이렇게 백업 되는 데이터가 언두 로그이다. 언두 로그는 트랜잭션의 롤백이 일어났을 때 데이터 복원을 위해 사용할 수 있고, 격리수 준에 따라 변경된 데이터를 읽지 않고 언두 로그에 있던 데이터를 반환하기도 한다.

### 5. 체인지 버퍼

InnoDB는 변경해야 할 인덱스 페이지가 버퍼 풀에 있으면 바로 업데이트를 수행하지만, 그렇지 않다면 임시 공간에 저장해두고 바로 사용자에게 결과를 반환한다. 이 때 사용하는 임시 메모리 공간이 체인지 버퍼이다.

참고로 유니크 인덱스는 체인지 버퍼를 사용할 수 없다. 결과를 전달하기 전에 반드시 중복 여부를 체크해야 하기 때문이다. 

### 6. 리두 로그

어떤 문제로 인해 서버가 비정상적으로 종료되었을 때 데이터 파일에 기록되지 못한 데이터를 잃지 않게 해주는 안전장치이다. 대부분의 데이터베이스 서버는 데이터 변경 내용을 로그로 먼저 기록한다. 쓰기 작업은 읽기 작업에 비해 비용이 큰 작업이기 때문에, 데이터베이스 서버는 쓰기 비용이 낮은 자료구조를 가진 리두 로그를 가지고 있다. 비정상 종료가 발생 시, 리두 로그를 이용해 직전의 상태로 복구한다. 아래와 같은 예시를 살펴보자.

1. 커밋됐지만 데이터 파일에 기록되지 않은 데이터

리두 로그에 저장된 데이터를 데이터 파일에 다시 복사하면 된다.

1. 롤백됐지만 데이터 파일에 이미 기록된 데이터

언두 로그의 내용을 가져와 데이터 파일에 복사한다.

2번의 경우에는 리두 로그가 전혀 필요하지 않은 것처럼 보이나, 최소한 변경이 커밋, 롤백, 트랜잭션 실행 중간이었는지를 확인하기 위해 리두 로그가 필요하다.

### 7. 어댑티브 해시 인덱스

사용자가 수동으로 생성하는 인덱스가 아닌 InnoDB 스토리지 엔진에서 사용자가 자주 요청하는 데이터에 자동으로 생성하는 인덱스이다. B-Tree 검색 시간을 줄여주기 위해 도입되었으며, 자주 읽히는 데이터 페이지의 키 값을 이용해 해시 인덱스를 만들고 필요할 때마다 해시 인덱스를 검색한다.

해시 인덱스는 ‘인덱스 키 값’과 해당 인덱스 키 값이 저장된 ‘데이터 페이지 주소’의 쌍으로 관리되며, 인덱스 키 값은 B-Tree 인덱스의 고유 번호와 B-Tree 인덱스의 실제 키 값 조합으로 생성된다.
