# 8장 인덱스

## 랜덤 I/O vs 순차 I/O

두 방식 모두 하드 디스크 드라이브의 플래터(원판)을 돌려서 읽어야 할 데이터가 저장된 위치로 디스크 헤더를 이동시킨 다음 데이터를 읽는 것을 의미한다. 3개의 페이지를 디스크에 기록하는 방식을 랜덤과 순차 방식으로 비교해보자.

순차 I/O는 3개의 페이지를 디스크에 기록하기 위해 시스템 콜을 1번 요청하고, 랜덤 I/O는 3번의 시스템 콜을 요청한다. 디스크에 기록하기 위해 순차 방식은 디스크의 헤드를 1번, 랜덤은 3번 움직이는 것이다. 디스크의 데이터를 쓰고 읽는데 걸리는 시간은 디스크 헤더를 움직이고 쓸 위치로 옮기는 단계에서 결정된다. 그렇다면 순차 방식이 랜덤 방식보다 빠르다고 볼 수 있다. 그러므로 랜덤 방식의 작업이 부하가 더 크다. DB의 대부분의 작업은 이런 작은 데이터를 빈번히 읽고 쓰기 때문에 MySQL 서버에서는 그룹 커밋이나 바이너리 로그 버퍼 또는 InnoDB 로그 버퍼 등의 기능이 내장돼 있다.

SSD는 디스크 원판을 가지지 않는다. 그러므로 SSD에서는 랜덤과 순차 방식의 차이가 없을 것 같지만, 실제로는 랜덤 방식이 순차 방식보다 전체 스루풋(Throughput)이 떨어진다. 그래서 SSD 사양에서도 순차 방식과 랜덤 방식의 성능 비교를 구분해서 명시한다.

## 인덱스 레인지 스캔 vs 풀 테이블 스캔

데이터를 읽기 위해 주로 랜덤 I/O를 사용, 풀 테이블 스캔은 순차 I/O를 사용한다. 큰 테이블의 레코드를 읽는 작업에서 인덱스를 사용하지 않고 풀 테이블 스캔을 유도할 때도 있다. 이는 순차 I/O가 랜덤 I/O보다 훨신 빨리 많은 레코드를 읽어올 수 있기 때문인데, 웹 서비스보다는 데이터 웨어하우스나 통계작업에서 많이 사용된다고 한다.

## 인덱스

칼럼(또는 칼럼들)의 값과 해당 레코드가 저장된 주소를 key-value쌍으로 만들어둔다. DBMS의 인덱스는 칼럼의 값을 주어진 순서로 정렬해서 보관한다. 

SortedList 자료구조는 항상 데이터를 정렬한다. 이들은 Insert Update Delete 작업이 일어나면 항상 데이터를 재위치시킨다는 단점이 있지만, 정렬되어 있기 때문에 원하는 데이터를 빠르게 가져올 수 있다. 쉽게 말해 Command 요청으로 인한 성능을 희생하고 **`Select의 성능을 높이는 기능`**이다.

인덱스를 역할에 따라 Primary Key와 Secondary Key로 구분할 수 있다.

1. Primary Key

레코드를 대표하는 칼럼의 값으로 만들어진 인덱스이다. PK는 테이블에서 레코드를 식별하는 기준이 되기 때문에 식별자라고 부르기도 한다.

1. Secondary Key

PK로 만들어진 인덱스가 아닌 나머지가 세컨더리 인덱스이다. 

데이터 저장 방식(알고리즘) 별로 구분하면 크게 2가지로 구분할 수 있다.

1. B-Tree

“Balanced”를 의미하며, 컬럼의 값을 변형하지 않고 원래 값을 이용해 인덱싱한다.

1. Hash Index

칼럼의 값으로 해시값을 계싼해서 인덱싱하며 매우 빠른 검색을 지원한다. 하지만 값을 변형하기 때문에 전방(Prefix) 일치와 같이 값의 일부만 검색하거나 범위를 검색할 때에는 사용할 수 없다.

## B-Tree 인덱스

최상위 노드가 루트 노드, 중간 노드가 브랜치 노드, 가장 하위에 있는 노드를 리프 노드라고 부른다. 데이터베이스에서 인덱스와 실제 데이터가 저장된 데이터는 따로 관리되는데, 인덱스의 리프노드는 실제 데이터 레코드를 찾아가기 위한 **주소값**을 가지고 있다.

참고로 인덱스의 키 값은 모두 정렬되어 있지만, 데이터 파일의 레코드는 정렬되어 있지 않고 임의의 순서대로 저장되있다. 데이터 파일의 레코드는 insert 순서대로 저장되는 것은 아니다. 레코드가 삭제되어 빈 공간이 생기면 DBMS는 공간을 재활용하도록 설계되어 있기 때문에  insert만 일어난다면 그럴 수 있어도, 항상 순서를 보장한다고 하면 틀린 얘기이다.

InnoDB 테이블에서 인덱스를 통해 레코드를 읽을 때는 데이터 파일로 바로 찾아가지 못한다. 인덱스에 저장되어 있는 프라이머리 키 값을 이용해 프라이머리 키 인덱스를 한 번 더 검색한 후, 프라이머리 키 인덱스의 리프 노드에 저장되어 있는 레코드를 읽는다. InnoDB 스토리지 엔진에서는 모든 세컨더리 인덱스 검색에서 데이터 레코드를 읽기 위해 프라이머리 키를 저장하고 있는 B-Tree를 다시 한 번 검색해야 한다.

## B-Tree 인덱스 키 추가 및 삭제

새로운 키 값이 B-Tree에 저장될 때 새로운 키 값이 즉시 인덱스에 저장될 수도 그렇지 않을 수도 있다. B-Tree에 저장될 때는 저장될 키 값을 이용해 적절한 위치를 검색하고, 위치가 결정되면 레코드의 키 값과 대상 레코드의 주소 정보를 리프노드에 저장한다. 리프 노드가 꽉 차서 더는 저장할 수 없다면 split해야 하는 상황이고 이는 상위 브랜치 노드까지 처리 범위가 넓어진다. 하여 B-Tree에서 쓰기 작업은 비용이 많이 드는 것으로 알려져 있다.

삭제는 리프 노드를 찾아 삭제 마크만 하면 작업이 완료된다. 삭제 마킹된 인덱스 공간은 계속 방치하거나, 재활용할 수 있다. InnoDB 엔진에서는 이 작업이 버퍼링되어 지연처리될 수 있다.

만약 B-Tree의 키 값을 변경한다면, 먼저 키 값을 삭제하고 다시 새로운 키 값을 추가하는 형태로 처리된다. 즉 앞서 설명했던 위의 두 작업을 실행하는 것이다.

## 인덱스 키 값의 크기

InnoDB 스토리지 엔진은 디스크에 데이터를 저장하는 가장 기본 단위를 페이지(Page) 또는 블록(Block)이라고 하며, 디스크의 모든 읽기 및 쓰기 작업의 최소 단위가 된다. 또한 페이지는 버퍼 풀에서 데이터를 버퍼링하는 기본 단위이다. 인덱스도 결국 페이지 단위로 관리되며, 루트와 브랜치, 리프 노드를 구분한 기준이 페이지 단위이다.

InnoDB 스토리지 엔진의 페이지 크기는 기본값이 16KB이다. 자식 노드 주소 영역을 12바이트로 가정하면, 하나의 인덱스 페이지(16KB)에 저장될 수 있는 키의 개수는 16 * 1024(16 + 12)이다.

인덱스의 키 값이 크면 디스크로부터 읽어야 하는 횟수가 늘어나고, 그만큼 느려짐을 의미한다.

## 읽어야 하는 레코드의 건수

인덱스를 통해 테이블의 레코드를 읽는 것은 인덱스를 거치지 않는 것보다 높은 비용이 드는 작업이다. 인덱스를 이용한 읽기의 손익분기점을 판단해야 하는데, 일반적인 DBMS의 옵티마이저는 인덱스를 통해 레코드를 1건 읽는 것을 그렇지 않은 것에 비해 4~5배 정도 비용이 더 많이 드는 작업으로 예측한다고 한다. 즉, 인덱스를 통해 읽어야 할 레코드의 건수가 전체 테이블 레코드의 20 ~ 25%가 넘으면 인덱스를 사용하는 것이 비효율적이다.

## 인덱스 레인지 스캔

검색해야 할 인덱스의 범위가 결정되었을 때 사용하는 방식이다. 검색하려는 값의 수나 검색 결과 레코드 건수와 관계없이 레인지 스캔이라고 표현한다.

인덱스의 리프 노드를 스캔하면서 실제 데이터 파일의 레코드를 읽어야 하는 경우가 있다. 이 경우는 리프 노드에 저장된 레코드 주소로 데이터 파일의 레코드를 읽어오는데, 인덱스 레인지 스캔은 데이터를 읽을 때마다 랜덤 I/O가 일어난다. 3건의 레코드가 검색 조건에 일치했다고 하면, 3번 필요한 것이다. 하여 인데스를 통해 데이터 레코드를 읽는 작업은 비용이 많이 드는 작업으로 분류할 수 있으며, 읽어야할 데이터 레코드가 20~ 25%를 넘으면 인덱스를 통해 읽는 것보다 데이터를 직접 읽는 것이 더 효율적인 처리 방식이 된다.

인덱스 레인지 스캔의 방식을 크게 3단계로 나눠보자.

1. 인덱스에서 조건을 만족하는 값이 저장되는 위치 찾기 → 인덱스 탐색
2. 탐색된 위치부터 필요한 만큼 인덱스를 읽는다 → 인덱스 스캔
3. 2번에서 읽어들인 인덱스 키와 레코드 주소를 이용하여 레코드가 저장된 페이지를 가져오고, 최종 레코드를 읽어온다.

위의 과정에서 1, 2번만 일어나는 경우가 있다. 이 경우를 **`커버링 인덱스`**라고 한다. 인덱스 만을 통해 다 가져올 수 있는 경우는 커버링 인덱스가 일어나며, 그렇지 않은 경우 3번의 단계를 거치게 된다.

## 인덱스 풀 스캔

인덱스를 사용하지만 처음부터 끝까지 모두 읽는 방식이다. 일반적으로 인덱스 레인지 스캔보다는 빠르지 않지만 테이블 풀 스캔보다는 효율적이다. 왜냐하면 인덱스에 포함된 칼럼만으로 쿼리를 처리할 수 있는 경우, 테이블의 레코드를 읽을 필요가 없기 때문이다. (커버링 인덱스)

## 루스 인덱스 스캔

느슨하게 또는 듬성듬성하게 인덱스를 읽는 것을 말한다. 인덱스 레인지 스캔과 비슷하게 작동하지만 중간에 필요치 않은 인덱스 키 값은 무시하고 다음으로 넘어가는 형태로 처리한다.

## 다중 컬럼 인덱스

두 개 이상의 컬럼으로 구성된 인덱스이다. 복합 컬럼 인덱스, Concatenated Index라고도 한다. 다중 컬럼 인덱스에서 인덱스 내에서 각 컬럼의 위치(순서)가 상당히 중요한데, 만약에 컬럼이 3개라고 하면 첫 번째 컬럼이 먼저 정렬이 되고, 두 번째 컬럼은 첫 번째 컬럼에 의존하여 정렬하고 세 번째 컬럼은 두 번째 컬럼에 의존하기 때문이다. 세 번째 컬럼의 조건으로 쿼리를 탐색하는 경우 직관적으로 세 번째 컬럼 기준으로 정렬되있는 것이 아닌, 앞의 두 컬럼에 의존하기 때문에 최악의 경우 인덱스의 맨 뒤 쪽에 위치할 수도 있다.

## B-Tree 인덱스의 정렬 및 스캔 방향

인덱스를 생성할 때, 정렬 규칙에 따라 오름차순 또는 내림차순으로 이루어진다. 즉, 거꾸로 읽으면 오름차순 내림차순 둘 다 가능하다. 따라서 어느 방향으로 인덱스를 읽을 지는 쿼리에 따라 옵티마이저가 실행 계획을 결정한다.

## 인덱스 스캔 방향

1. 오름차순 인덱스 : 작은 값의 인덱스 키가 B-Tree의 왼쪽으로 정렬
2. 내림차순 인덱스 : 큰 값의 인덱스 키가 B-Tree의 왼쪽으로 정렬
3. 인덱스 정순 스캔 : 인덱스 키의 크고 작음에 관계없이 인덱스 리프 노드의 왼쪽 페이지부터 오른쪽으로
4. 인덱스 역순 스캔 : 인덱스 키의 크고 작음에 관계없이 인덱스 리프 노드의 오른쪽 페이지부터 왼쪽으로

서비스의 요건에 맞게 어떤 정렬 순서의 인덱스를 선택해야 할지 결정해야 한다. 쿼리에 따라, 쿼리에서 자주 사용되는 정렬 순서대로 인덱스를 생성하는 것이 옳다고 판단되면 적용할 수 있다.

## 클러스터링 인덱스

테이블의 레코드를 비슷한 것(PK 기준)들끼리 묶어서 저장하는 형태로 구현되며, 주로 비슷한 값들을 동시에 조회한다는 점에서 착안했다고 한다. MySQL 기준 InnoDB에만 지원하며, 나머지 스토리지 엔진에서는 지원하지 않는다.

클러스터링 인덱스의 리프 노드에는 레코드의 모든 칼럼이 같이 저장되어 있다. 클러스터링 테이블은 그 자체가 하나의 거대한 인덱스 구조로 관리된다.

## 프라이머리 키가 세컨더리 인덱스에 미치는 영향

클러스터링되지 않은 테이블은 INSERT 작업 시 처음 저장된 공간에서 이동되지 않는다. 만약 세컨더리 인덱스가 실제 레코드가 저장된 주소를 가지고 있다면, 클러스터링 키 값이 변경될 때마다 데이터 레코드의 주소가 변경되고 그 때마다 해당 테이블의 모든 인덱스가 저장된 주소값을 변경해야 한다. 이런 오버헤드를 제거하기 위해 클러스터링 테이블의 모든 세컨더리 인덱스는 레코드의 주소가 아니라, PK 값을 저장하도록 구현되어 있다.

## 프라이머리 키는 AUTO-INCREMENT보다 업무적인 칼럼으로 생성

InnoDB 프라이머리 키는 클러스터링 키로 사용된다고 했다. 즉 PK로 검색하는 경우 훨씬 빠르게 처리할 수 있다는 것이다. PK는 의미적으로 중요한 역할을 하기 때문에 검색에서 상당히 빈번하게 사용된다. 설령 PK 칼럼의 크기가 크더라도, 업무적으로 해당 레코드를 대표할 수 있다면 그 칼럼을 프라이머리 키로 설정하는 것이 좋다.
