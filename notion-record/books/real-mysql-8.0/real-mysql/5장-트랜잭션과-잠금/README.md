# 5장 트랜잭션과 잠금

트랜잭션과 잠금

**잠금** : 동시성을 제어하기 위한 기능

**트랜잭션** : 데이터 정합성을 보장하기 위한 기능

**격리 수준** : 하나의 트랜잭션 내에서 또는 여러 트랜잭션 간의 작업 내용을 어떻게 공유하고 차단할 지 결정

## 트랜잭션 사용 시 주의사항

트랜잭션도 DBMS의 커넥션과 동일하게 필요한 최소의 코드에만 적용하는 것이 좋다. 특히 중간에 외부 api 호출이나, 아주 긴 작업(트랜잭션과 연관이 없는) 작업이 껴있다면 웹 서버 뿐 아니라 DBMS까지 위험해지는 상황이 발생한다.

프로그램의 코드에서 라인 수는 한두 줄이라고 하더라도 네트워크 작업이 있는 경우에는 반드시 트랜잭션에서 배제해야 한다.

## MySQL 엔진의 잠금

MySQL에서 사용되는 잠금은 MySQL 엔진 레벨과 스토리지 엔진 레벨로 구분할 수 있다. MySQL 엔진에서의 잠금은 모든 스토리지 엔진에 영향을 미치지만, 스토리지 엔진 레벨의 잠금은 스토리지 엔진 상호 간 영향을 미치지 않는다.

1. 글로벌 락

MySQL에서 제공하는 잠금 가운데 가장 범위가 크며, 한 세션에서 글로벌 락을 획득한 경우 다른 세션에서는 SELECT를 제외하고 대부분의 DDL, DML 문장을 실행하지 못하고 대기 상태로 남는다. MySQL 서버 전체에 영향을 미치기 때문에 작업 대상 테이블이나 데이터베이스가 다르더라도 동일하게 영향을 미친다.

1. 테이블 락

개별 테이블 단위로 설정되는 잠금이며, 명시적이나 묵시적으로 락을 획득할 수 있다. InnoDB 테이블의 경우 스토리지 엔진 차원에서 레코드 기반의 잠금을 제공하기 때문에 단순 데이터 변경 쿼리로는 묵시적 락을 설정하지 않는다. 대부분의 데이터 변경 쿼리에서는 무시되고, 스키마를 변경하는 쿼리(DDL)인 경우 영향을 미친다.

1. 네임드 락

GET_LOCK() 함수를 이용해 임의의 문자열에 대해 잠금을 설정한다. 많은 레코드에 대해 복잡한 요건으로 레코드를 변경하는 트랜잭션에 유용하게 사용할 수 있다. 동일 데이터를 변경하거나 참조하는 프로그램끼리 분류해서 네임드 락을 걸고 쿼리를 실행하면 효과가 좋다?

1. 메타데이터 락

데이터베이스 객체(테이블, 뷰 등)의 이름이나 구조를 변경하는 경우에 획득하는 잠금. 명시적으로 획득하거나 해제할 수 있는 것은 아니고 테이블의 이름을 변경하는 경우 등에 자동으로 획득하는 잠금.

## InnoDB 스토리지 엔진의 잠금

레코드 기반의 잠금 기능을 제공하며, 레코드 락 뿐만 아니라 레코드와 레코드 사이의 간격을 잠그는 갭락도 존재한다.

1. 레코드 락

InnoDB에서는 인덱스의 레코드를 잠근다. 레코드 자체가 아니라, 인덱스의 레코드를 잠그는 것이다.

1. 갭 락

레코드 자체가 아니라 레코드와 인접한 레코드 사이의 간격만을 잠그는 것. 역할은 레코드와 레코드 사이에 새로운 레코드가 생성되는 것을 제어하는 것

1. 넥스트 키 락

레코드 락과 갭락을 합쳐 놓은 형태의 잠금.

## 인덱스와 잠금

InnoDB의 잠금은 레코드를 잠그는 것이 아니라 인덱스를 잠그는 방식으로 처리된다고 했다. 변경해야할 레코드를 찾기 위해, **`검색한 인덱스의 레코드를 모두 락을 걸어야 한다`**. 적절한 인덱스가 설정되있는 경우 범위를 줄여서 락을 걸 수 있지만, 그렇지 않은 경우 테이블 풀스캔으로 모든 레코드에 대해 잠금을 걸어야 한다. 그러므로 MySQL InnoDB에서 인덱스 설계는 중요한 영역이다.

가령 아래와 같은 상황을 예시로 든다.

1번 트랜잭션이 update 후 commit을 하지 않음

2번 트랜잭션이 update 작업을 수행하려 함

3번 트랜잭션이 update 작업을 수행하려 함

세 트랜잭션이 모두 동일한 record를 update하고 있다고 할 때, 2번과 3번 트랜잭션은 1번 트랜잭션이 commit할 때까지 레코드에 대해 대기를 해야한다. 1번 트랜잭션이 락을 걸고 있기 때문에, 2 3번은 계속해서 대기하게 된다.

1. READ UNCOMMITED

Commit 여부와 상관없이 각 트랜잭션에서의 변경 내용이 다 보인다. 어떤 트랜잭션에서 처리한 작업이 완료되지 않았는데도 보인다 → DIRTY READ

DIRTY READ, NON-REPEATABLE READ, PHANTOM READ
