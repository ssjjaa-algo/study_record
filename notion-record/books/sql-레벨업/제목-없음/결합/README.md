# 크로스 결합

- 데카르트 곱 연산이므로, 6개의 레코드와 4개의 레코드를 가진 조합의 연산 수 = 6 * 4
- 결합 연산 중 가장 비용이 많이 든다.
- 아래와 같은 코드를 만들면, 크로스 결합을 실수로 사용한 형태
    - 결과값이 엄청나게 많으니 실무에서도 거의 안쓴다.

```java
SELECT * FROM Employees ,Deparyments
```

# 내부 결합(inner join)

- 데카르트 곱의 **`부분 집합`**

### 내부 결합과 같은 기능을 하는 상관 서브쿼리

- 일반적으로 상관 서브쿼리를 사용해 대체 가능한 경우가 많다.

```java
SELECT E.emp_id, E.emp_name, E.dept_id,
	(SELECT D.dept_name
		FROM Departments D
		WHERE E.dept_id = D.dept_id) AS dept_name
	FROM Employees E
```

- dept_id를 부서 테이블의 기본 키라고 하면, 기본 키를 이용하여 상관 서브쿼리를 스칼라 서브쿼리로 사용할 수 있다.
- 내부 결합 vs 상관 서브쿼리
    - 내부 결합이 WIN, 상관 서브쿼리를 스칼라처럼 사용하는 경우 결과 레코드 수만큼 서브쿼리 실행

# 외부 결합(outer join)

- 데카르트 곱의 부분 집합이 아님
- 경우에 따라 부분 집합이기도 한다
- left, right, full outer

# Hash

- 입력에 대해 어느 정도 유일성과 균일성을 가진 값을 출력하는 함수
- 일단 작은 테이블을 스캔하고, 결합 키에 해시 함수를 적용해서 해시 함수로 변환
- 이어서 다른 테이블을 스캔, 결합 키가 해시값에 존재하는지 확인

<aside>
💡 해시 테이블을 작은 테이블을 이용해서 만드는 이유는, 해시 테이블이 **워킹 메모리**에 만들어지기 때문

</aside>

### 언제 사용?

- Nested Loops에서 적절한 구동 테이블이 존재하지 않거나, Nested Loops의 성능이 좋지 않음
- Nested Loops의 내부 테이블에 인덱스 없을 때

<aside>
💡 쉽게 말해, **Nested Loops가 효율적이지 못하는 경우.**

</aside>

- OLTP가 높은 경우에, Hash를 사용하면 DBMS 메모리를 많이 사용하기 때문에 피해야 한다.

# Sort

- Merge join이라고 부르기도 한다.
- 결합 대상 테이블들을 각각 결합 키로 정렬하고, 일치하는 결합 키를 찾으면 결합
- 각각 정렬한다는 것에 포인트를 두면,
    - 정렬 과정이 수반되어 메모리 사용량 많음
    - 역시 메모리 사용이 많이 되기 때문에, OLTP가 높다면 **TEMP 탈락** 가능성 존재
- 동치, 부등호 사용 가능, 부정 조건에는 사용 불가

### 언제 사용?

- 테이블 정렬을 생략할 수 있는 경우 (이미 정렬이 되어 있는 경우)
- 그 외에는 일단 Nested Loops, Hash를 우선적으로 고려하라.
