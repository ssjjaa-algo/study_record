# 예시

- 두 개의 Pod가 있고 각각 IP가 있다고 가정
- 서로 통신을 위한 방법 2가지
    - `/etc/hosts` 파일에 엔트리를 추가
        - 첫 번째 Pod의 `/etc/hosts` → `web`은 `10.244.2.5`
        - 두 번째 Pod의 `/etc/hosts` → `test`는 `10.244.1.5`
        - 수천 개의 파드..수동으로 하면 많아서 불편하다.
    - **`중앙 DNS 서버`**
        - 각 Pod가 DNS 서버를 바라보도록 `/etc/resolv.conf`에 엔트리를 추가한다.
        - 즉, nameserver가 DNS 서버 IP를 가리키도록 한다
        - DNS 서버의 IP는 `10.96.0.1`이라고 가정
    - 새 Pod가 생성될 때마다 DNS 서버에 해당 Pod의 레코드를 추가해 다른 Pod들이 새 Pod에 접근할 수 있게 한다.
    - 새 Pod의 `/etc/resolv.conf`도 DNS 서버를 바라보도록 구성해서, 새 Pod가 클러스터의 다른 Pod들을 해석할 수 있게 한다.

## Kubernetes의 DNS 서버

- Kubernetes는 클러스터 내부에 DNS 서버를 배포하는 방식으로 DNS를 구현
    - Kubernetes 1.12 이전에는 이 DNS 서버가 **kube-dns**
    - Kubernetes 1.12부터는 DNS 서버가 **CoreDNS**
- DNS 서버는 Kubernetes 클러스터의 `kube-system` 네임스페이스에 **Pod로 배포**
- 보통 두 개의 Pod로 배포(Deployment)
- 이 Pod는 **coredns 실행 파일**을 실행
    - CoreDNS는 설정 파일이 필요 → **`Corefile`**
    - `/etc/coredns/Corefile`

```bash
.:53 {
	errors
	health
	
	kubernetes **cluster.local** in-addr.arpa ip6.arpa {
		pods insecure
		upstream
		fallthrough in-addr.arpa ip6.arpa
	}
	prometheus :9153
	proxy . /etc/resolv.conf
	cache 30
	reload
}	
```

- CoreDNS가 Kubernetes와 연동되어 동작하도록 하는 핵심 플러그인은 **kubernetes 플러그인**이다.
    - 클러스터의 최상위 도메인(top level domain)을 설정 → `cluster.local`
    - 따라서 CoreDNS가 관리하는 모든 레코드는 이 도메인 아래에 속한다.
- `pods` 옵션은 클러스터에서 Pod에 대한 레코드를 생성
    - Pod 레코드는 IP를 특정 포맷으로 바꿔 만들어진다. (dot을 -로)
    - 이 기능은 기본으로 꺼져 있지만, Corefile에서 해당 엔트리를 통해 켤 수 있다

## 외부 도메인 포워딩

Pod가 `www.google.com`을 해석하려고 하면,

- CoreDNS는 이를 Pod의 `/etc/resolv.conf`에 지정된 nameserver로 포워딩
    - Pod의 `/etc/resolv.conf`는 Kubernetes 노드의 nameserver를 사용하도록 설정되어 있다.
- Corefile은 ConfigMap으로 전달됨
    - CoreDNS Pod에 **ConfigMap 오브젝트로 전달된다.**
    - 따라서 설정을 변경해야 한다면 ConfigMap을 수정하면 된다.

## CoreDNS

- Kubernetes 클러스터
- Pod 또는 Service가 생성될 때마다 DNS 서버 내부 데이터베이스에 레코드를 추가
- Pod들이 CoreDNS를 바라보도록 해야함
    - Pod가 DNS 서버에 접근할 때 어떤 주소를 사용할까?
    - DNS 솔루션을 배포하면, CoreDNS를 클러스터 내부 다른 컴포넌트들이 사용할 수 있도록 **Service도 같이 생성**

    ```bash
    kubectl get service -n kube-system
    NAME      TYPE        CLUSTER-IP  EXTERNAL-IP   PORT(S)
    kube-dns  ClusterIP   10.96.0.10  <none>        53/UDP,53/TCP
    ```

    - Service 이름은 **`kube-dns`**
    - 기본적으로 이 서비스의 IP가 Pod들의 `/etc/resolv.conf`에 nameserver로 설정
    - Pod가 생성될 때 **`kubelet`**이 자동으로 DNS 설정을 해준다.

    ```bash
    cat /etc/resolv.conf
    nameserver    10.96.0.10 # dns 서버
    ```

- Pod가 올바른 nameserver를 가지면, 이제 다른 Pod/Service를 해석할 수 있다
- 예를 들어 web service
    - `web-service`
    - `web-service.default`
    - `web-service.default.svc.cluster.local`
- `nslookup web-service`나 `host web-service`는 완전한 FQDN이 반환
    - `web-service.default.svc.cluster.local`
    - `/etc/resolv.conf`에 **search 엔트리**가 있기 때문
        - `default.svc.cluster.local`
        - `svc.cluster.local`
        - `cluster.local`

```bash
cat /etc/resolv.conf
search default.svc.cluster.local svc.cluster.local cluster.local
```

- search 엔트리는 **service에 대한 것만** 포함
- Pod에 접근하려면 **Pod의 전체 FQDN을 명시해야 한다.**
