# Service

- 어떤 Pod가 다른 Pod에서 제공하는 서비스를 이용해야 한다면, 일반적으로 **Service**를 사용
- 오렌지 Pod를 블루 Pod가 접근 가능하도록 하려면,
    - 오렌지 Service를 만든다
    - 오렌지 Service에는 IP 주소와 이름이 할당된다
    - 블루 Pod는 오렌지 Service의 IP나 이름으로 오렌지 Pod에 접근할 수 있다
- Service는 생성되면, Pod가 어떤 노드에 있든 상관없이 **클러스터의 모든 Pod에서 접근 가능**
    - Service는**`클러스터 전체`**에 걸친 개념, 특정 노드에 묶여 있지 않음
- Service가 클러스터 내부에서만 접근 가능하다 → **`ClusterIP`**
- 클러스터 밖에서도 접근하고 싶다 → **`NodePort`**

---

# 상황

- 3개의 노드 안에 아무것도 없는 상태라고 가정
- 각 노드에는 kubelet 프로세스가 실행되고 있음
    - kubelet은 Pod 생성 책임을 가짐
    - kubelet은 kube-apiserver를 통해 클러스터 변경 사항을 감시
    - 새 Pod가 생성되어야 하면 노드에 Pod를 만들고 CNI 플러그인을 호출하여 Pod 네트워크를 구성
- kube-proxy
    - kube-apiserver를 통해 클러스터 변경 사항을 감시
    - 새 Service가 생성되면 kube-proxy가 동작
- Service는 실제로 존재하지 않는다 → **`가상 오브젝트`**
    - 프로세스, 네임스페이스, 인터페이스가 없다
- Kubernetes에서 Service 오브젝트를 생성하면, Service는 **미리 정해진 IP 범위에서 IP 하나를 할당받음**
    - 그리고 각 노드에서 실행 중인 kube-proxy는 이 Service IP를 확인하고, **각 노드에 포워딩 규칙(forwarding rules)** 을 만듬
    - 이 Service IP로 들어오는 트래픽은 실제 Pod IP로 보내라는 규칙을 노드마다 만들어줌
- 클러스터 어디에서든 Pod가 Service IP로 접근할 때 그 트래픽이 실제 Pod IP로 전달됨
- Service가 생성/삭제될 때마다 kube-proxy는 이 규칙을 생성/삭제

# kube-proxy

- kube-proxy는 여러 모드를 지원
- user space 모드
    - kube-proxy가 각 Service마다 포트를 listen하고 프록시로 Pod에 연결
- ipvs 모드
    - ipvs 규칙을 생성해서 프록시 처리
- iptables 모드 **`(default)`**
    - iptables 규칙 생성

# 예시

- Node1에 `DB`라는 Pod가 있고 IP가 `10.244.1.2`
    - 이 Pod를 클러스터 내부에 제공하기 위해 ClusterIP 서비스를 만든다
    - Service가 생성되면 Kubernetes가 Service에 IP를 할당한다
    - 예로 `10.103.132.104`할당
- Service IP는 kube-apiserver의 옵션에서 지정한 범위에서 나온다

    ```bash
    kube-api-server --service-cluster-ip-range ipNet (Default: 10.0.0.0/24)
    
    ps aux | grep kube-api-server
    kube-apiserver--authorization-mode=Node,RBAC--service-cluster-ip-range=10.96.0.0/12
    
    ```

    - `-service-cluster-ip-range`
        - 실습문제 → What is the range of IP addresses configured for PODs on this cluster?

        ```bash
        cat /etc/kubernetes/manifests/kube-apiserver.yaml   | grep cluster-ip-range
        ```

    - 기본값이 `10.0.0.0/24`
    - 위의 예시의 경우 Service IP 범위는 `10.96.0.0` ~ `10.111.255.255`
- Pod 네트워크를 구성할 때, Pod 네트워크 CIDR로 `10.244.0.0/16`을 지정
    - 그러면 Pod IP 범위는 `10.244.0.0` ~ `10.244.255.255`
- Pod IP 대역과 Service IP 대역은 서로 **`겹치면 안 된다`**

# iptables 규칙 확인

- kube-proxy가 만든 규칙은 iptables NAT 테이블에서 확인

```bash
iptables –L –t net | grep db-service

KUBE-SVC-XA5OGUC7YRHOS3PU   tcp -- anywhere 10.103.132.104  /* default/db-service: cluster IP */ tcp dpt:3306
DNAT                        tcp -- anywhere anywhere        /* default/db-service: */ tcp to:10.244.1.2:3306
KUBE-SEP-JBWCWHHQM57V2WN7   all -- anywhere anywhere        /* default/db-service: */
```

- 목적지가 `10.103.132.104:3306`(Service IP:Port)인 트래픽은
- 목적지를 `10.244.1.2:3306`(Pod IP:Port)으로 변경하라
- NodePort 타입 Service를 만들면 kube-proxy는 iptables 규칙을 추가로 생성
    - 모든 노드의 특정 포트로 들어오는 트래픽을
    - 백엔드 Pod의 포트로 전달
