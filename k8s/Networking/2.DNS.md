## 두 시스템 A, B와 이름 해석(Name Resolution)

- A와 B가 `192.168.1.1`과 `192.168.1.11`로 할당
- 시스템 B의 IP를 매번 외우는 대신, B에게 `db`라는 이름
    - 앞으로는 IP 대신 `db`라는 이름으로 시스템 B를 ping
- 시스템 A의 **/etc/hosts** 파일에 엔트리를 추가
    - IP 주소와 이름을 적어둔다

```go
$ cat >> /etc/hosts
192.168.1.11        db
```

- 이후 ping db

### /etc/hosts

- 호스트 A는 `/etc/hosts` 파일에 적힌 내용을 **그대로** 받아들임
    - 하나의 IP에 여러 name이 붙을 수 있다
    - 예를 들어 google.com을 192.168.1.11로도 설정
        - (db, google.com) 두 개 설정되어 있는 상황
        - google.com을 쳤을 때 우리가 아는 google이 아니라 매핑된 곳으로 이동해버림
- 하나의 시스템에 대해 여러 개의 이름을 붙일 수 있음
    - `db`
    - `*google.com*`
- `/etc/hosts` 파일에는 원하는 만큼 서버에 대해 원하는 만큼 이름을 넣을 수 있다.
- 호스트 A에서 ping, ssh, 혹은 시스템 안의 어떤 애플리케이션/도구가 다른 호스트를 이름으로 참조하면 먼저 `/etc/hosts`를 보고 그 호스트 이름의 IP를 찾음
- 이처럼 **호스트 이름을 IP로 변환하는 것**을 **`name resolution(이름 해석)`** 이라고 한다.

---

## /etc/hosts의 어려움

- 소규모에서는 용이할 수 있으나 대규모에서는 어렵다
    - 일일이 수기로 하면서 변경되면 다 업데이트 하고 추가해줘야 하고.. 관리할 게 많아진다는 것
- 이런 엔트리를 한 곳에서 중앙 관리하는 서버로 옮기자
    - 그 서버가 **`DNS 서버`**
    - 각 호스트는 `/etc/hosts` 대신 **DNS 서버에 질의**해서 이름을 IP로 해석하도록 설정

---

## DNS 서버

- DNS 서버의 IP가 `192.168.1.100`이라고 하자.
- 각 호스트에는 DNS 해석 설정 파일이 있다.
- `/etc/resolv.conf`

```bash
cat /etc/resolv.conf
nameserver 192.168.1.100 
```

- 호스트가 어떤 이름을 만났는데 로컬에 없으면 DNS 서버에 물어봄
    - 어떤 호스트의 IP가 바뀌면 DNS 서버만 업데이트
    - 다른 호스트들은 앞으로 자동으로 새 IP를 해석
- 그렇다고 `/etc/hosts`가 완전히 필요없어진 것은 아님
    - 예시) 개인용 테스트를 위한 서버 추가 (DNS 서버에는 추가하지 않고)
- 둘 다 존재할 경우의 우선순위
    - 기본적으로는 /etc/hosts부터 보고 DNS 서버를 봄
    - 순서 수정 가능 → `/etc/nsswitch.conf`
    - /etc/nsswitch.conf
        - `files` → `/etc/hosts`
        - `dns` → DNS 서버
- 내부 DNS에도 없고 /etc/hosts에도 없는 외부 도메인은?
    - 내부 DNS 서버 자체에서 “모르는 도메인”을 외부 공개 DNS로 **forward(포워딩)** 하도록 설정 가능

## 도메인 이름과 계층 구조

- **Top Level Domain**
    - `.com` : 상업/일반 목적
    - `.net` : 네트워크 관련
    - `.edu` : 교육 기관
    - `.org` : 비영리 조직
- `.` : 루트(root)
- `.com` : TLD
- `google` : 구글에 할당된 도메인
- `www` : 서브도메인(subdomain)
- 서브도메인은 google 아래에서 기능별로 더 그룹화할 수 있다.
    - `maps.google.com` : 지도 서비스(서브도메인 `maps`)
    - `drive.google.com` : 스토리지 서비스(서브도메인 `drive`)
    - `mail.google.com` : 이메일 서비스(서브도메인 `mail`)

---

## 도메인 요청 보낼 때 과정

- 요청은 먼저 조직의 내부 DNS 서버로
    - 내부 DNS는 `apps`나 `google` 같은 것을 모르므로 인터넷으로 요청을 포워딩
- 인터넷에서는 여러 DNS 서버를 거쳐 최종 IP를 알아낸다
    - 어떤 DNS 서버는 다음 단계의 DNS 서버를 알려주고
    - 또 다른 DNS 서버는 구글 DNS로 안내하고
    - 구글 DNS는 해당 서비스(예: maps)의 IP를 제공한다
- 일정 기간동안 캐싱 가능

## /etc/resolv.conf의 search 옵션(짧은 이름으로 접근하고 싶을 때)

- `web.mycompany.com`
- 원칙적으로는 `web.mycompany.com`으로 ping해야 한다.
    - 회사 내부에서는 `web`이라고만 쳐도 `web.mycompany.com`을 의미하게 만들고 싶다.
- `/etc/resolv.conf`에 `search` 엔트리를 추가한다.

```
search mycompany.com
```

- 여러 search 도메인을 줄 수도 있다

```
search mycompany.com prod.mycompany.com
```

- `web`을 찾을 때 `web.mycompany.com`, `web.prod.mycompany.com` 등 여러 후보를 시도한다.

## nslookup, dig

- nslookup, dig은 DNS 서버에 호스트 이름을 질의
    - **nslookup은 로컬 `/etc/hosts` 엔트리를 고려하지 않음**
- dig이 더 세부 사항을 보여줌
