## CNI에서 정의한 책임

- 컨테이너 네트워크 네임스페이스 생성
- 해당 네임스페이스를 올바른 네트워크에 연결
- 적절한 네트워크 플러그인을 호출해 네트워크 구성
- Kubernetes의 경우, 컨테이너를 생성하는 주체는 **컨테이너 런타임**
    - 따라서 CNI 플러그인을 호출하는 것도 컨테이너 런타임의 역할
- CNI 플러그인은 컨테이너 생성 이후 호출되어야 함
    - 플러그인을 호출하는 주체는 컨테이너를 생성하는 컴포넌트
    - 컨테이너 생성 책임을 가진 컴포넌트는 **컨테이너 런타임**
        - 예시
        - containerd
        - CRI-O

---

## CNI 플러그인 위치

```
ls /opt/cni/bin

bridge dhcp falnnel host-local ipvlan ...
```

- 이 디렉토리를 확인하면 다음과 같은 실행 파일
    - bridge
    - dhcp
    - flannel
    - 기타 CNI 플러그인들
- 플러그인 선택과 설정은 다음 디렉토리에서 정의

```
ls /etc/cni/net.d/

10-bridge.conflist
```

- 컨테이너 런타임은 이 디렉토리의 설정 파일을 읽어 어떤 플러그인을 사용할지 결정한다
- 만약 여러 파일이 있다면 알파벳 순서로 가장 먼저 오는 파일을 사용한다

## 예: bridge 설정 파일

예를 들어 bridge 설정 파일은 다음과 같은 형식이다.

```json
{
  "cniVersion": "0.3.1",
  "name": "mynet",
  "type": "bridge",
  "bridge": "cni0",
  "isGateway": true,
  "ipMasq": true,
  "ipam": {
    "type": "host-local",
    "subnet": "10.22.0.0/16",
    "routes": [
      { "dst": "0.0.0.0/0" }
    ]
  }
}
```

이 형식은 CNI 표준에서 정의한 플러그인 설정 파일 형식이다.

---

## 주요 설정 항목 설명

### `"type": "bridge"`

사용할 플러그인 타입이 bridge임을 의미한다.

---

### `"isGateway": true`

브리지 인터페이스에 IP를 할당하여 게이트웨이로 동작하도록 설정한다.

---

### `"ipMasq": true`

IP Masquerade(NAT)를 설정할지 여부를 정의한다.

---

### `"ipam"` 섹션

IPAM(IP Address Management) 설정을 정의한다.

여기에서 다음을 지정한다:

- 사용할 IP 대역(subnet)
- Pod에 할당될 IP 범위
- 필요한 라우팅 정보

```json
"type": "host-local" // IP 주소를 이 노드에서 로컬로 관리
"type": "dhcp" // 외부 DHCP 서버
```

---

## 동작 흐름 요약

1. Kubernetes가 컨테이너 생성 요청
2. 컨테이너 런타임(containerd 또는 CRI-O)이 컨테이너 생성
3. 런타임이 `/etc/cni/net.d` 설정 확인
4. 설정 파일에 정의된 플러그인 이름 확인
5. `/opt/cni/bin`에서 해당 플러그인 실행
6. 플러그인이 네트워크 구성 수행

---

## 정리

- CNI 플러그인은 `/opt/cni/bin`에 위치
- 어떤 플러그인을 사용할지는 `/etc/cni/net.d`에서 결정
- 설정 파일은 CNI 표준 JSON 형식
- bridge, flannel 등 다양한 플러그인 존재
