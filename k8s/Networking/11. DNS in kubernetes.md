## 클러스터 내부 DNS 해석

- Pod와 Service 같은 클러스터 구성요소 간 DNS 해석을 말하는 것
- 3노드 Kubernetes 클러스터가 있고, 그 위에 몇 개의 Pod와 Service가 배포되어 있다고 가정
    - 각 노드에는 노드 이름과 IP가 있다.
    - 노드 이름과 IP는 조직의 DNS 서버에 등록되어 있을 수도 있다.

## Kubernetes의 기본 DNS 서버

- Kubernetes는 클러스터를 설정하면 기본적으로 **`내장 DNS 서버`**를 배포
    - 만약 Kubernetes를 수동으로 설치했다면, 이 부분은 직접 구성해야 함
- 모든 Pod와 Service가 각각의 IP를 갖고 서로 통신할 수 있다고 가정한다.

## 예시

- test Pod가 있고 IP는 `10.244.1.5`
- web Pod가 있고 IP는 `10.244.2.5`
- Pod들이 IP로 서로 통신 가능한 상태라고 가정
- web 서버를 test Pod에서 접근 가능하게 하기 위해 Service(web-service)를 만듬
    - Service는 IP를 하나 할당받는다. 예를 들어 `10.107.37.188`

Service가 생성되면 Kubernetes DNS 서비스는 그 Service에 대한 **`DNS 레코드`**를 만든다.

- Service 이름 → Service IP 형태
- 이제 클러스터 내부에서 어떤 Pod든 Service 이름으로 그 Service에 도달

---

## 같은 네임스페이스에 있는 경우

- 같은 네임스페이스 안에서는 서로를 이름만으로 부를 수 있다
- 현재 예시에서는 test Pod, web Pod, Service가 모두 같은 네임스페이스(기본 `default`)에 있다
- 따라서 test Pod에서 Service를 **서비스 이름만으로** 접근할 수 있다
    - `web-service`

    ```bash
    curl http://web-service
    ```


## 다른 네임스페이스에 있는 경우

- web-service가 `apps`라는 다른 네임스페이스에 있다고 가정
- default 네임스페이스에서 접근하려면
    - `web-service.apps`

    ```bash
    curl http://web-service.apps
    ```

## Service DNS

- DNS 서버는 네임스페이스마다 서브도메인을 하나 만든다.
- 네임스페이스의 모든 Pod와 Service는 그 네임스페이스 이름의 서브도메인 아래로 묶인다.
- 모든 Service는 `svc`라는 서브도메인 아래로 묶인다.
    - `web-service` : 서비스 이름
    - `apps` : 네임스페이스 이름(네임스페이스별 서브도메인)
    - `svc` : 서비스 그룹을 묶는 서브도메인
- 모든 Service와 Pod는 클러스터의 루트 도메인으로 묶임
    - 기본 루트 도메인은 `cluster.local`
- 따라서 서비스의 FQDN(완전한 도메인 이름)
    - `web-service.apps.svc.cluster.local`

```bash
curl http://web-service.apps.svc.cluster.local
```

## Pod DNS 레코드

- Pod에 대한 레코드는 기본적으로는 생성되지 않는다
- Pod 레코드를 만들 때는 Pod 이름을 그대로 쓰지 않는다
    - Kubernetes는 각 Pod의 **`IP를 기반`**
        - IP 주소의 점(`.`)을 대시(`-`)로 바꾼 문자열을 호스트 이름으로 사용
- 타입은 `pod`
- 루트 도메인은 항상 `cluster.local`
- 예를 들어 default 네임스페이스의 Pod IP가 `10.244.2.5`
    - IP의 점을 대시로 바꾼 이름을 생성하고
    - namespace는 `apps`
    - type은 `pod`
    - root는 `cluster.local`

```bash
curl http://10-244-2-5.apps.pod.cluster.local
```
