- 네트워킹 부분을 하나의 독립된 프로그램으로 분리
    - 예시) 브리지 네트워크를 구성하는 프로그램을 만들고 이름을 **bridge**라고 가정
    - 이 프로그램은 컨테이너를 브리지 네트워크에 연결하기 위해 필요한 모든 작업을 수행
    - bridge는 각기 다른 컨테이너 런타임의 환경을 모두 고려해야한다
        - 어떤 인자를 받아야 할까?
        - 어떤 명령을 지원해야 할까?
        - 컨테이너 런타임이 이 프로그램을 어떻게 호출해야 할까?
        - 어떻게 해야 Kubernetes나 Rocket이 이 프로그램을 올바르게 호출할 수 있을까?
    - 이런 부분을 표준화해야 한다.
- 그래서 등장한 것이 **Container Networking Interface(CNI)**

---

## CNI

- CNI는 컨테이너 런타임 환경에서 네트워킹 문제를 해결하기 위한 프로그램을 어떻게 만들어야 하는지 정의한 **표준**
    - 이 프로그램들을 **plugin(플러그인)** 이라고 부른다.
- CNI는 다음을 정의
    - 플러그인은 어떤 형태로 작성되어야 하는가
    - 컨테이너 런타임은 플러그인을 어떻게 호출해야 하는가

---

## CNI가 정의하는 책임 분담

- 이 표준을 따르면

> 어떤 컨테이너 런타임이든
> 
> 
> 어떤 네트워크 플러그인이든
> 
> 서로 함께 동작할 수 있다.
> 

### 컨테이너 런타임의 책임

1. 각 컨테이너마다 네트워크 네임스페이스 생성
2. 컨테이너가 연결될 네트워크 식별
3. 컨테이너 생성 시 플러그인을 `ADD` 명령으로 호출
4. 컨테이너 삭제 시 플러그인을 `DEL` 명령으로 호출
5. 네트워크 설정을 JSON 파일로 전달

### 플러그인의 책임

- 플러그인은 다음 명령을 지원
    - `ADD`
    - `DEL`
    - `CHECK`
- 파라미터 받아야 함
    - container ID
    - network namespace
    - 기타 네트워크 관련 정보
- 작업 수행
    - 컨테이너(또는 Pod)에 IP 주소 할당
    - 필요한 라우트 설정
    - 컨테이너 간 통신 가능하도록 네트워크 구성
- 결과를 정해진 형식(JSON)으로 반환

---

## CNI 기본 플러그인

- bridge
- vlan
- ipvlan
- macvlan
- Windows용 플러그인
- IPAM 플러그인 (host-local, DHCP 등)
- 서드 파티 플러그인
    - Weave
    - Flannel
    - Cilium
    - VMware NSX
    - Calico
    - Infoblox 등
- 모두 CNI 표준을 구현하고 있기 때문에 CNI를 지원하는 런타임과 함께 사용할 수 있음

---

## Docker는 예외

- Docker는 자체 네트워크 표준을 사용
    - **CNM (Container Network Model)**
- Docker에서 직접 CNI 플러그인을 지정해서 실행할 수는 없다.

```bash
docker run --network=cni=bridge nginx # 불가
```

---

- Docker와 CNI를 완전히 함께 사용할 수 없는 것은 아님
- 예시
    1. Docker 컨테이너를 네트워크 설정 없이 생성하고
    2. CNI 플러그인을 수동으로 호출해 네트워크를 구성

```bash
docker run --network=none nginx
bridge add 2e34dcf34 /var/run/netns/2e34dcf34
```

- Kubernetes가 Docker를 사용할 때 이런 방식으로 동작
    - Docker로 컨테이너를 생성
    - 설정된 CNI 플러그인을 호출
    - 네트워크 구성을 완료
