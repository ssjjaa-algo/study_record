## Docker는 로컬 파일 시스템에 데이터를 어디에 저장하는가

- 기본적으로 Docker는 다음 경로 아래에 데이터를 저장
    - `/var/lib/docker`
- 그 아래에는 다음과 같은 여러 폴더
    - `aufs`
    - `containers` (컨테이너 관련 저장)
    - `image` (이미지 관련 저장)
    - `volumes`
    - 등등

---

## Docker는 이미지/컨테이너 파일을 어떻게 저장하는가

- Docker의 **layered architecture(레이어 구조)**
- Docker는 이미지를 빌드할 때 레이어 형태로 만듬
- Dockerfile에서 **각 instruction 한 줄이 하나의 layer를 생성**
- 각 layer는 이전 layer 대비 **변경된 내용만 저장**
    1. 첫 번째 레이어: base ubuntu 배포판
    2. 두 번째 레이어: apt 패키지 설치
    3. 세 번째 레이어: Python 패키지 설치
    4. 네 번째 레이어: 소스 코드 복사
    5. 다섯 번째 레이어: entrypoint 업데이트
- 각 레이어는 변경분만 저장하기 때문에 크기에도 반영

---

## layer architecture

- a 어플과 b 어플의 일정 부분이 같고, 일정 부분이 다른 경우
    - 동일한 레이어는 다시 빌드하지 않고 캐시에 있는 기존 레이어를 재사용한다
    - **`다른 부분만 새로 만든다`**
- 이를 통해
    - 이미지를 더 빠르게 빌드
    - 디스크 공간을 효율적으로 사용
- 애플리케이션 코드를 업데이트 하는 경우
    - 코드만 바뀌면 Docker는 이전 레이어를 캐시에서 재사용하고, 최신 소스 코드 레이어만 다시 만듬
- 빌드가 완료된 뒤에는 레이어의 내용을 수정할 수 없음
- 수정하려면 docker build로 새로운 이미지를 만든다

## 컨테이너 실행 시 writable layer 생성

- `docker run`으로 이미지 기반 컨테이너를 실행
    - Docker는 이미지 레이어 위에 **새로운 writable layer를 만듬**
        - writable layer는 컨테이너가 생성하는 데이터를 저장
            - 로그, 컨테이너가 생성한 임시파일, 수정한 파일 등등
        - 컨테이너가 살아있는 동안만 유지된다. 변경 사항 유지 x
    - 컨테이너에서 temp.txt를 만듬 → 이 layer에 저장

---

## 이미지 레이어의 파일을 수정하려면?

- 이미지 레이어의 파일은 read-only이므로 원칙적으로는 수정할 수 없음
    - 예를 들어 애플리케이션 코드는 이미지에 “구워 넣어” 빌드했으므로 이미지 레이어에 포함되며, 컨테이너 실행 후에는 read-only
- 컨테이너 안에서 소스 코드를 수정해서 테스트하는 것은 불가능한가? 가능하다.
    - 어떻게?
        - Docker는 파일을 저장하려는 시점에 자동으로 그 파일을 writable layer로 복사
        - 이후 수정은 writable layer에 있는 복사본에 대해 수행
        - **`copy-on-write`** 메커니즘

## Docker volume 생성 및 마운트

- 데이터 유지를 위해 Persistence Volume를 이용해야함

```bash
docker volume create data_volume
```

- 이 명령은 Docker 호스트의 다음 경로 아래에 `data_volume` 폴더를 만듬
    - `/var/lib/docker/volumes/data_volume`
- 컨테이너 실행 시 `-v` 옵션으로 이 볼륨을 컨테이너의 writable layer 내부 경로에 마운트

```bash
docker run -v data_volume:/var/lib/mysql mysql
```

- 새 컨테이너가 생성되고
- 호스트에 만든 volume이 컨테이너의 `/var/lib/mysql`에 마운트
- 결과적으로 데이터베이스가 쓰는 데이터는 사실상 Docker 호스트에 만든 volume에 저장
- 컨테이너가 삭제되어도 데이터는 남음
- volume을 미리 만들지 않으면?
    - 자동으로 볼륨을 만들고 컨테이너에 마운트한
    - `/var/lib/docker/volumes` 에서 생성된 볼륨 확인 가능
- 이 방식은 Docker가 만든 volume을 `/var/lib/docker/volumes` 아래에서 마운트하는 것이므로 **`volume mounting`**이라고 함

---

## bind mount

- 만약 데이터가 이미 호스트의 다른 위치에 존재하는 경우
    - 예를 들어 Docker 호스트에 외부 스토리지가 `/data`에 마운트되어 있고
    - MySQL 데이터를 거기에 저장하고 싶은 경우

```bash
docker run -v /data/mysql:/var/lib/mysql mysql
```

- 호스트의 임의 경로를 컨테이너에 직접 연결하므로 **bind mounting**이라고 부름

**`정리하면 mount는 두 가지`**

- volume mount: `/var/lib/docker/volumes` 아래 Docker volume을 마운트
- bind mount: 호스트의 아무 디렉터리나 지정해서 마운트

---

## v 대신 --mount 사용 (권장)

- -v는 구옵션
- `-mount`는 key=value 형식으로 더 명확하게 작성

```bash
docker run --mount type=bind,source=/data/mysql,target=/var/lib/mysql mysql
```

- type: bind
- source: 호스트 경로
- target: 컨테이너 경로

## 누가 이런 동작을 담당하는가? (Storage Driver)

- 레이어 아키텍처 유지, writable layer 생성, copy-on-write를 위한 파일 이동 등 이런 모든 동작을 수행하는 것은 **`storage driver`**
    - aufs
    - btrfs
    - devicemapper
    - overlay
    - overlay2
- 어떤 storage driver를 쓰는지는 기반 OS에 따라 달라짐
