1. 관리자가 Persistent Volume(PV)을 생성한다.
2. 사용자가 Persistent Volume Claim(PVC)을 생성한다.
3. Pod에서 PVC를 사용해 스토리지를 마운트한다.
- 이 방식은 **`Static Provisioning(정적 프로비저닝)`**

---

## Dynamic Provisioning의 필요성

> 애플리케이션이 스토리지를 요청하면
>
>
> 자동으로 디스크가 생성되고 연결되도록 하는 것
>
- **`Storage Class`**
    - 어떤 provisioner를 사용할지 정의하고
    - PVC가 생성될 때 자동으로 디스크를 생성
- 이 방식은 **`Dynamic Provisioning(동적 프로비저닝)`**

# Storage Class 생성

- Storage Class는 별도의 Kubernetes 오브젝트다.

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
provisioner: kubernetes.io/gce-pd
```

- `provisioner`: 사용할 스토리지 프로비저너 지정
    - 여기서는 GCE Persistent Disk (예시)

---

# 기존 구조와 비교

- 기존 구조:
    - Pod → PVC → PV → 실제 디스크
- Storage Class 사용 시:
    - Pod → PVC → (StorageClass가 자동으로 PV + 실제 디스크 생성)
- **PVC는 여전히 생성하지만 PV는 자동으로 생성된다**

---

# PVC에서 Storage Class 사용

- PVC 정의에 `storageClassName`을 정의

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-claim
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: standard
  resources:
    requests:
      storage: 5Gi
```

- PVC가 생성되면
- Kubernetes는 `standard` StorageClass를 확인
- 해당 provisioner를 사용해 GCP에 5Gi 디스크를 생성
- 자동으로 PV 생성
- PVC와 PV 바인딩

# 다양한 Provisioner

- AWS EBS
- Azure Disk
- Azure File
- CIFS
- Portworx
- ScaleIO
- 기타 CSI 기반 드라이버
- 각 provisioner는 추가 파라미터를 받을 수 있음
    - 예: Google Persistent Disk의 경우
        - disk type: standard / SSD
        - replication mode: none / regional-pd

# 다양한 Storage Class 구성 예

- **`Storage Class`**라고 불리는 이유
    - 서로 다른 성능과 특성을 가진 스토리지를 여러 등급(class)으로 정의할 수 있기 때문
    - 사용자는 PVC를 만들 때 원하는 스토리지 등급을 선택

### Silver (일반 디스크)

```yaml
metadata:
  name: silver
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-standard
  replication-type: none
```

### Gold (SSD)

```yaml
metadata:
  name: gold
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd
  replication-type: none
```

### Platinum (SSD + replication)

```yaml
metadata:
  name: platinum
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd
  replication-type: regional-pd
```
