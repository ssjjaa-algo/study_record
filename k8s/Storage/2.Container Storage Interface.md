## 역할

- CRI, CNI는 각각 컨테이너, 네트워킹의 여러 솔루션을 지원하기 위해 나온 역할
    - 이와 비슷하게 CSI 역시 **`여러 저장소를 지원하기 위한 것`**
        - Portworx
        - Amazon EBS
        - Azure Disk
        - Dell EMC Isilon / PowerMax
        - Unity
        - XtremIO
        - NetApp
        - Nutanix
        - HPE
        - Hitachi
        - Pure Storage
- **CSI는 Kubernetes 전용 표준이 아님**
    - CSI는 범용(universal) 표준을 목표로 하며, CSI를 구현하면 **어떤 컨테이너 오케스트레이션 도구든** 지원되는 플러그인을 통해 **어떤 스토리지 벤더든** 연동할 수 있음
    - K8s가 아닌 다른 오케스트레이션 도구도 상관없다

## CSI 표준 흐름

- 오케스트레이터가 호출할 **RPC(Remote Procedure Call)** 집합을 정의
    - 스토리지 드라이버는 이 RPC들을 구현해야함
- Pod가 생성되면서 볼륨이 필요해지면,
    - 오케스트레이터(여기서는 Kubernetes)가 **CreateVolume RPC** 를 호출
    - 이때 볼륨 이름 같은 세부 정보를 함께 전달
    - 스토리지 드라이버는 이 RPC를 구현하여 요청을 처리
        - 스토리지 어레이에 새 볼륨을 프로비저닝한 뒤
        - 결과를 반환
- 볼륨을 삭제해야 하면,
    - 오케스트레이터가 **DeleteVolume RPC** 를 호출
    - 스토리지 드라이버는 이 RPC를 구현하여
        - 스토리지 어레이에서 해당 볼륨을 해제
- CSI 스펙에는 아래 내용이 정의
    - 호출자가 어떤 파라미터를 보내야 하는지
    - 드라이버(솔루션)가 무엇을 받아야 하는지
    - 어떤 에러 코드들을 주고받아야 하는지
- https://github.com/container-storage-interface/spec
    - CSI 전용 스펙 깃허브 페이지
