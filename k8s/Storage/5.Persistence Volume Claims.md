## PV와 PVC 관계

- Persistent Volume과 Persistent Volume Claim은 Kubernetes에서 **서로 다른 두 개의 오브젝트**
    - **관리자(Admin)** → Persistent Volume들을 미리 생성
    - **사용자(User)** → Persistent Volume Claim을 생성하여 스토리지 요청
- PVC가 생성되면 Kubernetes는 요청 조건에 맞는 PV를 찾아 자동으로 바인딩

---

## 바인딩 동작 원리

- PVC가 생성되면 Kubernetes는 다음 조건을 기준으로 PV를 찾음
    - 충분한 capacity
    - access modes 일치
    - volume mode
    - storage class
    - 기타 속성

### 특징

- 하나의 PV는 하나의 PVC와만 **`1:1`**로 바인딩
    - PV가 1GB이고 PVC가 500MB를 요청해도, 나머지 500MB를 다른 PVC가 사용할 수 없다 → **`1:1`**
- 더 작은 PVC도 조건이 맞으면 더 큰 PV에 바인딩될 수 있다.
- 적합한 PV가 없으면 PVC는 `Pending` 상태
- 이후 새로운 PV가 생성되면 자동으로 바인딩
- 특정 PV를 사용하고 싶다면 labels와 selectors를 사용해 명시적으로 바인딩

---

## PVC 생성

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-claim
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 500Mi
```

## PVC 삭제 시 PV는 어떻게 되는가?

- PV의 **`Reclaim Policy`** 에 따라 달라짐

### Retain (기본값)

- PVC 삭제 후에도 PV는 남아 있음
- 관리자가 수동으로 삭제해야 함
- 다른 PVC에서 재사용 불가

### Delete

- PVC 삭제 시 PV도 자동 삭제
- 클라우드 스토리지(EBS 등)도 함께 삭제

### Recycle (Deprecated)

- 데이터 삭제 후 재사용
- 내부적으로 작은 Pod를 실행해 `rm -rf` 방식으로 파일 삭제
- 아래와 같은 이유로 Deprecated 되었다고 한다
    - 안전한 데이터 삭제 보장 안 됨
    - 스냅샷 처리 안 됨
    - 메타데이터 처리 안 됨
    - 일부 플러그인에서만 동작
    - inode 메타데이터 잔존 가능
    - 권한 문제 발생 가능
