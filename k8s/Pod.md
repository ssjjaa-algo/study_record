## **쿠버네티스 포드(Pods) 상세 정리**

쿠버네티스에서 가장 기본이 되는 실행 단위인 포드(Pod)의 개념과 특징, 그리고 다중 컨테이너 환경에 대한 상세 내용

### **1. 기본 개념**

- **포드의 정의:** 쿠버네티스에서 생성할 수 있는 **가장 작은 오브젝트**
    - 애플리케이션의 **단일 인스턴스**를 의미
    - 쿠버네티스는 워커 노드에 컨테이너를 직접 배포하지 않고, **컨테이너를 포드라는 객체로 캡슐화**하여 배포함

---

### **2. 포드의 확장(Scaling) 방식**

애플리케이션 사용자가 증가하여 확장이 필요한 경우의 작동 원리

- **수평 확장:** 부하를 분산하기 위해 기존 포드 내에 새 컨테이너를 추가하는 것이 아니라, **동일한 애플리케이션 인스턴스를 가진 새로운 포드를 추가**로 생성함
- **노드 확장:** 현재 노드의 용량이 부족하면 클러스터에 새 노드를 추가하고 그 위에 포드를 배포함
- **관계:** 일반적으로 포드와 애플리케이션 컨테이너는 **1:1 관계**를 유지함
- **요약:** 확장 시에는 새 포드 생성, 축소 시에는 기존 포드 삭제를 통해 조절함

---

### **3. 다중 컨테이너 포드 (Multi-container Pods)**

하나의 포드에 여러 컨테이너가 포함되는 특수한 경우

- **제한 사항:** 일반적으로 동일한 종류의 컨테이너를 여러 개 넣지는 않음
- **헬퍼(Helper) 컨테이너:** 웹 애플리케이션 컨테이너를 보조하는 데이터 처리, 로그 수집 등의 태스크를 수행하는 컨테이너를 함께 배치 가능
    - 즉 **생애주기를 같이해야 하는 어떤 컨테이너가 있다면** 같이 배포할 수도 있다
- **특징 및 장점:**
    - **생명 주기 공유:** 메인 앱 컨테이너와 헬퍼 컨테이너는 함께 생성되고 함께 소멸함
    - **네트워크 공유:** 동일한 네트워크 네임스페이스를 사용하므로 서로 `localhost`로 직접 통신 가능
    - **스토리지 공유:** 동일한 저장 공간을 쉽게 공유하여 사용 가능
- **참고:** 다중 컨테이너 포드는 드문 사례이며, 본 과정에서는 주로 1포드 1컨테이너 방식을 다룸

---

### **4. 도커 단독 실행 vs 쿠버네티스 포드**

쿠버네티스가 제공하는 자동화 관리의 이점

- **도커 단독 방식:** 앱 컨테이너와 헬퍼 컨테이너 간의 네트워크 링크 수동 설정, 공유 볼륨 수동 생성, 앱 소멸 시 헬퍼 컨테이너 수동 삭제 등 모든 관리를 직접 해야 함
- **쿠버네티스 방식:** 포드 내 컨테이너 구성을 정의하기만 하면 저장소/네트워크 공유 및 생명 주기 관리를 쿠버네티스가 자동으로 수행함
- 단순한 단일 컨테이너 앱이라도 포드로 감싸 실행하면 향후 아키텍처 변경이나 확장에 유리함

---

### **5. 포드 배포 및 확인 방법**

### **포드 배포 (Kube control run)**

- `kubectl run nginx --image=nginx` 명령어 실행
  - nginx라는 이름을 가지고, image가 nginx인 포드를 배포한다는 의미
- **동작 원리:** 자동으로 포드를 생성하고 지정된 이미지(예: nginx)를 도커 허브 또는 지정된 프라이빗 저장소에서 다운로드하여 배포함

### **상태 확인 (Kube control get pods)**

- `kubectl get pods` 명령어로 클러스터 내 포드 목록 조회
- **상태 변화:** `ContainerCreating` (생성 중) → `Running` (실행 중)

## YAML 기반 포드(Pod) 정의 및 생성

---

### 1. 쿠버네티스 정의 파일의 4대 필수 루트 필드

모든 쿠버네티스 정의 파일은 다음 4가지 최상위 필드를 반드시 포함해야 함

- **apiVersion:** 오브젝트 생성을 위해 사용하는 쿠버네티스 API 버전
    - 포드 생성 시에는 `v1` 사용
    - 기타 값: **apps/v1, extensions/v1beta1** 등
- **kind:** 생성하고자 하는 오브젝트의 유형
    - 예: **Pod, ReplicaSet, Deployment, Service** 등
- **metadata:** 이름(name), 라벨(labels) 등 오브젝트 자체에 대한 데이터
    - 딕셔너리(Dictionary) 형태이며 하위 항목은 오른쪽으로 들여쓰기 처리
- **spec:** 생성할 오브젝트에 대한 실제 상세 명세(Specification)
    - 오브젝트 유형에 따라 제공해야 하는 정보가 상이함

---

### 2. Metadata 상세 및 들여쓰기 규칙

메타데이터는 오브젝트를 설명하는 데이터를 담고 있으며 작성 시 엄격한 구조 준수 필요

- **구조적 특징:**
    - name과 labels는 metadata의 자식 요소이므로 들여쓰기 필수
    - 형제 요소(Siblings)인 name과 labels는 동일한 수준의 들여쓰기를 가져야 함
    - 잘못된 예: labels가 name보다 더 들어가 있으면 name의 자식으로 인식되어 오류 발생
- **필드 내용:**
    - **name:** 포드의 이름(문자열) 설정 (예: `myapp-pod`)
    - **labels:** 임의의 키-값 쌍을 가지는 딕셔너리
        - 목적: 수백 개의 포드 중 프론트엔드, 백엔드, 데이터베이스 등을 그룹화하고 나중에 필터링하기 위함
    - **제약:** `metadata` 바로 아래에는 정해진 필드(name, labels 등)만 올 수 있으나, `labels` 내부에는 사용자가 원하는 어떤 키-값 쌍도 추가 가능

---

### 3. Spec 상세 (포드 및 컨테이너 설정)

포드가 실제로 어떤 컨테이너를 가질지 정의하는 섹션

- **containers:** 포드 내의 컨테이너들을 정의하는 **리스트(List) 또는 배열(Array)** 형태
    - 포드는 다중 컨테이너를 가질 수 있으므로 리스트 형식을 채택
- **리스트 작성법:**
    - 하위 항목 앞에 -를 붙여 리스트의 첫 번째 아이템임을 표시
    - 각 아이템은 딕셔너리 형태로 `name`과 `image` 속성을 가짐
    - **image:** 도커 허브 등 저장소에서 가져올 이미지 이름 (예: `nginx`)

---

### 4. 포드 정의 YAML 예시

```yaml
#pod-definition.yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
  labels:
    app: myapp
    type: front-end
spec:
  containers: ## multi container가 가능하기 때문에 containers인 점 확인
  - name: nginx-container
    image: nginx
```

---

### 5. 주요 관리 명령어 및 확인

- **오브젝트 생성:** `kubectl create -f pod-definition.yaml`
    - `f` 옵션 뒤에 작성한 파일 경로를 입력하여 생성 수행
- **목록 조회:** `kubectl get pods`
    - 현재 클러스터에서 가동 중인 포드 목록 확인
- **상세 정보 확인:** `kubectl describe pod myapp-pod`
    - 포드 생성 시간, 할당 라벨, 내부 컨테이너 정보 및 관련 **이벤트(Events)** 내역 상세 확인 가능
